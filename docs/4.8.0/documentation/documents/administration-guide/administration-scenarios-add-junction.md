# Добавление нового ответвления

## Проблематика  
Администраторам необходимо гибко настраивать маршрутизацию в IAM Proxy для интеграции с новыми сервисами и поддержки нескольких контуров безопасности. Отсутствие стандартизированного подхода к добавлению ответвлений может привести к ошибкам конфигурации, нарушению безопасности или сбоям в работе аутентификации и проксирования. Важно соблюдать соответствие актуальной схеме конфигурации (`rds-api-configuration.schema.json`) при добавлении новых `junction`.

Дополнительную информацию о доступных параметрах и их значениях для конфигурации ответвлений подробнее в разделе [Параметры ответвлений](../installation-guide/installation.md#заполнение-раздела-с-описанием-параметров-ответвлений).

## Предусловия  
1. **Доступ к среде**:  
   - наличие прав администратора в системе управления конфигурациями или на сервере с IAM Proxy;  
   - доступ к файлу конфигурации IAM Proxy (локально или через UI);  
   - возможность перезагрузить конфигурацию через API или развертывание.  

2. **Требуемые навыки**:  
   - понимание структуры `proxy_jct_list` и соответствия полей JSON-схеме `rds-api-configuration.schema.json`;  
   - знание обязательных полей: `junctionPoint`, `junctionName`;  
   - опыт работы с YAML и инструментами валидации (например, `yamllint`, `jsonschema`).  

3. **Инструменты**:  
   - текстовый редактор (VS Code, Vim и др.);  
   - утилиты проверки YAML/JSON;  
   - доступ к терминалу или API IAM Proxy для применения изменений.

## Правила эксплуатации  
1. **Безопасность конфигураций**:  
   - все изменения должны проходить через утвержденный процесс (GitOps, CI/CD или защищенный UI);  
   - запрещено вносить правки напрямую в production без предварительного тестирования в staging-среде.  

2. **Форматы данных**:  
   - конфигурационные файлы должны быть валидными по YAML-синтаксису и соответствовать ограничениям из `rds-api-configuration.schema.json`;  
   - `junctionPoint` должен быть уникальным в рамках одного профиля развертывания;  
   - поля `junctionName` и `junctionPoint` являются обязательными.  

3. **Особенности настройки**:  
   - перед добавлением убедитесь, что целевые `serverAddresses` доступны, разрешаются через DNS и защищены TLS (если `https: true`);  
   - при использовании `applyJctRequestFilter` убедитесь, что указанные фильтры существуют в системе (например, `set-header-host-to-backend`, `large-upload`);  
   - при настройке `healthcheck` проверьте, что 'бэкенд отвечает на указанный `path` с допустимыми HTTP-статусами.

## Последовательность выполнения

### Способ 1: Через внешнюю систему управления конфигурациями
1. **Открыть UI системы управления конфигурациями**:  
   - выберите артефакт, соответствующий профилю развертывания IAM Proxy для нужного стенда.

2. **Добавить новое ответвление**:  
   - найдите секцию `proxy_jct_list` (или аналогичную, отвечающую за список junction);  
   - создайте новый элемент с обязательными и опциональными параметрами:

     ```yaml
     proxy_jct_list:
       - junctionName: "Новое приложение"
         junctionPoint: /new-app
         indexUrl: /new-app/index.html
         https: true
         sslCommonName: ".backend.mycompany.ru"
         transparent: false
         serverAddresses:
           - "app-server.mycompany.ru:8443"
         applyJctRequestFilter: "set-header-host-to-backend"
         limitRequests: 100
         authorizeByRoleTemplate: "App_NewUser"
         healthcheck:
           enabled: true
           path: /health
           interval: 15000
           timeout: 5000
           fall: 3
           rise: 2
           validStatus: "200,204,302"
     ```

3. **Сохранить и применить изменения**:  
   - подтвердите и сохраните конфигурацию;  
   - дождитесь синхронизации с IAM Proxy или инициируйте принудительную перезагрузку конфигурации.

### Способ 2: Через YAML-файл (вручную)
1. **Редактировать конфигурационный файл**:  
   - откройте файл `ansible/inventories/<StandX>/group_vars/all/main.yml`;  
   - добавьте новое ответвление в список `proxy_jct_list`, соблюдая синтаксис и ограничения схемы:

     ```yaml
     proxy_jct_list:
       - junctionName: "Внутренний сервис"
         junctionPoint: /internal-api
         serverAddresses:
           - "api-v3.corp.local:443"
         https: true
         transparent: true
         applyJctRequestFilter: "large-upload"
     ```

2. **Применить изменения**:  
   - сохраните файл;  
   - перезагрузите конфигурацию IAM Proxy:

     ```shell
     curl -X POST https://iam-proxy.mycompany.ru/actuator/reload -k
     ```

## Результат  
Новое ответвление успешно добавлено и доступно через IAM Proxy. Проверьте:  
- доступность сервиса:  
  ```shell
  curl -k https://iam-proxy.mycompany.ru/new-app/
  ```  
- отсутствие ошибок в логах IAM Proxy (`/opt/iamproxy/logs/error.log`);
- корректное отображение в UI IAM Proxy (если используется);
- статус healthcheck — убедитесь, что бэкенд помечен как `up`.

## Описание механизмов безопасности
1. **Ограничение доступа**:
   - редактирование конфигураций разрешено только пользователям с правами администратора;
   - доступ к UI и API защищен с использованием TLS 1.2+ и аутентификации по OIDC.

2. **Целостность данных**:
   - конфигурационные файлы хранятся в защищенных хранилищах (Vault, SecMan);
   - перед применением выполняется валидация по схеме `rds-api-configuration.schema.json`;
   - пример безопасной конфигурации:

     ```yaml
     proxy_jct_list:
       - junctionName: "Админ-панель"
         junctionPoint: /admin
         https: true
         sslCommonName: "*.secure.mycompany.ru"
         serverAddresses:
           - "admin-01.secure.mycompany.ru:8443"
         applyJctRequestFilter: "set-header-host-to-backend"
         authorizeByRoleTemplate: "regex:^Admin_.*"
         healthcheck:
           enabled: true
           path: /status
     ```

3. **Логирование и аудит**:
   - все изменения конфигурации фиксируются в централизованном журнале (Platform V Audit SE);
   - события включают: пользователя, время, IP, старые и новые значения;
   - рекомендуется настроить оповещения о добавлении новых junction через MONA.
