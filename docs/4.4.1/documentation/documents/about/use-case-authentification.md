# Пройти аутентификацию

**Основной сценарий**

1. Не аутентифицированный пользователь обращается к ресурсу через IAM Proxy.
2. IAM Proxy перенаправляет пользователя в провайдер аутентификации для выполнения аутентификации.
3. Пользователь вводит данные необходимые для аутентификации (логин, пароль, сертификат, OTP и т.п.).
4. Провайдер аутентификации аутентифицирует пользователя, и перенаправляет обратно на IAM Proxy.
5. IAM Proxy перенаправляет пользователя на ресурс с шага 1.
6. Аутентифицированный пользователь обращается к ресурсу (
   см. [Получить ресурс требующий аутентификации](use-case-accessing-resource.md)).

**Альтернативный сценарий**

1. Не аутентифицированный пользователь обращается к IAM Proxy указав данные необходимые для аутентификации в HTTP
   заголовке `Authorization`, чтобы использовать аутентификацию direct-auth.
2. IAM Proxy делает запрос в провайдер аутентификации на получение токенов, и передает HTTP заголовок `Authorization`.
3. Провайдер аутентификации аутентифицирует пользователя, и отправляет токены в ответе.
4. IAM Proxy сохраняет токены в локальном кэше, и получает данные с запрошенного ресурса. 
5. Пользователь получает запрошенный ресурс от IAM Proxy.

**Исключительный сценарий**

1. Не аутентифицированный пользователь обращается к ресурсу через IAM Proxy.
2. IAM Proxy перенаправляет пользователя в провайдер аутентификации для выполнения аутентификации.
3. Пользователь вводит некорректные данные необходимые для аутентификации.
4. Провайдер аутентификации сообщает об ошибке. Не удается аутентифицировать пользователя.
