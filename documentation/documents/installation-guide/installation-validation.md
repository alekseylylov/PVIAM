# Чек-лист валидации установки

После окончания развертывания IAM Proxy, необходимо проверить корректность развертывания, как самого IAM Proxy, так и
его компонентов.

*В IAM Proxy реализованна проверка компонентов на этапе развертывания, в случае ошибки, развертывание останавливается и
возвращает пользователю сообщение об ошибке.*

## Предусловие

Для успешного прохождения чек-листа, пользователь должен обладать следующими правами и доступами:

1. Доступ в IAM Proxy. Наличие учетной записи в IDP, на которую настроен IAM Proxy.
2. Доступ в PACMAN (CFGA) с правами редактирования (администрирования) для артефакта RDS Server.

## Чек-лист валидации установки RDS Server

### Цель выполнения
> Проверка корректности установки и работы RDS Server необходима для подтверждения его интеграции с IAM Proxy и 
функциональности сервисов (StandIn, Аудит, Конфигуратор). Выполняется при сомнении в стабильности развертывания или 
после обновления компонентов.

### Последовательность действий
1. Перейти на UI RDS Server:
   - для виртуальных машин: Открыть `https://<host:port>/rds-for-proxy/`;
   - для OpenShift: Через route IAM Proxy, предварительно настроив проксирование до RDS Server.
2. Проверить статусы сервисов (StandIn, Аудит, Конфигуратор).

### Проверка результата
Вывод статусов должен содержать:
```text
state=OK
errorMessage=[опциональное описание]
timestamp=<дата>
```
Пример успешного ответа:
```text
Статус сервиса Конфигуратор
state=OK
errorMessage=Used Priority reader
timestamp=2023/05/29 10:52:18
```


## Чек-лист валидации установки RDS Client

### Цель выполнения
> Проверка взаимодействия RDS Client с RDS Server через PACMAN (CFGA) подтверждает корректную передачу данных между
компонентами. Выполняется при изменении конфигурации или интеграционных тестах.

### Последовательность действий
1. Войти в PACMAN (CFGA) с правами администратора.
2. Выбрать артефакт RDS Server и версию.
3. Изменить параметр `junctionName` в произвольном ответвлении.
4. Подождать завершения передачи данных.
5. Проверить обновление имени ответвления на стартовой странице IAM Proxy.

### Проверка результата
Имя измененного ответвления должно отображаться на стартовой странице IAM Proxy без ошибок.

## Чек-лист валидации установки IAM Proxy

### Цель выполнения
> Подтверждение работоспособности IAM Proxy как центрального компонента аутентификации. Выполняется после развертывания
или при сбоях в доступе.

### Последовательность действий
1. Перейти на точку входа IAM Proxy (`/proxy/`).
2. Пройти аутентификацию через форму (логин/пароль).
3. Нажать `ОК`.

### Проверка результата
Открытие стартовой страницы IAM Proxy подтверждает успешную аутентификацию и корректную работу.

## Чек-лист валидации установки Сервиса ограничения (квотирования) входящих запросов (SRLS)

### Цель выполнения
> Проверка корректности интеграции IAM Proxy с SRLS для ограничения количества входящих запросов по заданным квотам.  
Выполняется при настройке защиты от DDoS-атак или тестировании политик квотирования.

### Последовательность действий
1. Войти в UI SRLS.
2. Создать правило квотирования для домена IAM Proxy (например, `platform.mycompany.ru`).
3. Установить минимальную квоту (например, 5 запросов в минуту).
4. Использовать `curl` или Postman для отправки множества запросов к IAM Proxy:
     ```shell
     for i in {1..10}; do curl -k https://platform.mycompany.ru/proxy/; done
     ```
   - Убедиться, что после превышения квоты возвращается статус `429 Too Many Requests`.

5. В UI SRLS выполнить поиск событий по типу `rate_limiting`.
6. Убедиться, что отображаются записи о блокировке запросов.
7. В файле `platformauth.log` найти строки с `Rate limit exceeded`.
8. Убедиться, что они соответствуют времени блокировки запросов.

### Проверка результата
- **Успешный сценарий:**  
  При превышении квоты возвращается статус `429`, а в логах SRLS и IAM Proxy отображаются события ограничения.
  
- **Пример успешного ответа:**  
  ```http
  HTTP/1.1 429 Too Many Requests
  Content-Type: application/json
  X-RateLimit-Limit: 5
  X-RateLimit-Remaining: 0
  ```

- **Пример записи в логах SRLS:**
  ```json
  {
    "timestamp": "2024-04-05T12:34:56Z",
    "source": "SRLS",
    "event_type": "rate_limiting",
    "domain": "platform.mycompany.ru",
    "ip_address": "192.x.x.100",
    "action": "blocked"
  }
  ```
