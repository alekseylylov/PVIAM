## Аппаратное обеспечение

Расчет потребности в физическом оборудовании происходит на основе коэффициентов повторной подписки, применяемых к
требованиям по виртуальным ресурсам для каждого из стендов.

Для установки компонента/продукта в k8s требуется следующая конфигурация аппаратного обеспечения 
(рекомендуемая конфигурация на которой проводилось НТ):

| Название компонента | ПО среды функционирования                                | Количество | Список контейнеров                                     | CPU (количество ядер) | ОЗУ (МБ) | Внутренние диски (ГБ) | Горизонтальное масштабирование |
|:--------------------|:---------------------------------------------------------|:-----------|--------------------------------------------------------|:----------------------|:---------|:----------------------|:-------------------------------|
| auth-egressgateway  | Kubernetes/Platform V DropApp (опционально OpenShift NS) | 2 pod      | istio-proxy, vault-agent                               | 0.9                   | 640      | -                     | да                             |
| auth-iamproxy       | Kubernetes/Platform V DropApp (опционально OpenShift NS) | 4 pod      | iamproxy, vault-agent, fluent-bit-sidecar, istio-proxy | 3.55                  | 3800     | -                     | да                             |
| auth-ingressgateway | Kubernetes/Platform V DropApp (опционально OpenShift NS) | 2 pod      | istio-proxy, vault-agent                               | 3.9                   | 1152     | -                     | да                             |

При горизонтальном масштабировании продукта для подтверждения соответствия ожидаемым нефункциональным характеристикам,
рекомендуется предварительно провести нагрузочное тестирование на целевой конфигурации конкретной инсталляции.

Минимальная конфигурация элементов развертывания компонентов в k8s, при использовании SSM и OPM
(максимальная производительность 20 RPS, при среднем времени отклика более 143 ms):

| Pod name                 | Container name           | Count | CPU request | CPU limit | MEM request | MEM limit | Start (sec) | Summ CPU | Summ MEM |
|--------------------------|--------------------------|-------|-------------|-----------|-------------|-----------|-------------|----------|----------|
| auth-egressgateway       | istio-proxy              | 1     | 100         | 100       | 128         | 128       | 5           | 200      | 256      |
|                          | vault-agent              |       | 100         | 100       | 64          | 128       | 5           |          |          |
| auth-iamproxy            | iamproxy                 | 1     | 200         | 200       | 256         | 512       | 69          | 600      | 1168     |
|                          | fluent-bit-sidecar       |       | 200         | 200       | 200         | 400       | 68          |          |          |
|                          | istio-proxy              |       | 100         | 100       | 128         | 128       | 68          |          |          |
|                          | vault-agent              |       | 100         | 100       | 64          | 128       | 57          |          |          |
| auth-ingressgateway      | istio-proxy              | 1     | 100         | 100       | 128         | 256       | 5           | 200      | 384      |
|                          | vault-agent              |       | 100         | 100       | 64          | 128       | 4           |          |          |
| **Итого**                |                          |       | 1000        | 1000      | 1032        | 1808      |             |          |          |

**Примечание:**  
Для обеспечения корректной работы всех компонентов в рамках одного Namespace рекомендуется выделить **не менее 1 CPU (1000 милиядер)**
и **2 ГБ оперативной памяти (1808 МБ)**, исходя из суммарных потребностей контейнеров, указанных в таблице.
Эти значения соответствуют минимальным требованиям для запуска всех сервисов с учетом их ресурсных ограничений и запросов.

Минимальные размеры разделов жесткого диска на VM:

| Раздел             | Для разработки | Для прома |
|--------------------|----------------|-----------|
| /usr               | 6 ГБ           | 6 ГБ      |
| /opt               | 8 ГБ           | 16 ГБ     |
| /opt/iamproxy/logs | 1 ГБ           | 16 ГБ     |
| /root              | 8 ГБ           | 8 ГБ      |
| /home              | 4 ГБ           | 4 ГБ      |
| /tmp               | 1 ГБ           | 2 ГБ      |
| /var               | 4 ГБ           | 8 ГБ      |

Для установки IAM Proxy на VM требуется предварительно выделить сервера под следующие компоненты (количество с
георезервированием по двум зонам, минимальная конфигурация позволяет запустить сервис и выполнять 
не более 1 запроса в секунду без ограничений по времени выполнения запроса, рекомендуемая конфигурация 
рассчитана не более чем на 10 запросов секунду без ограничений по времени выполнения запроса, некоторые 
компоненты не обязательны и их состав зависит от варианта развертывания, установка компонентов не входящих в 
поставку IAM Proxy производится по отдельной документации этих компонентов):

| Компонент                                  | Рекомендуемая конфигурация (CPU/MEM/HDD) | Минимальная конфигурация (CPU/MEM/HDD) | Минимальное количество экземпляров в PROD |
|--------------------------------------------|------------------------------------------|----------------------------------------|-------------------------------------------|
| Platform V IAM, IAM Proxy (AUTH)           | VM x86 (2/8/150)                         | VM x86 (1/2/70)                        | 2                                         |
| Platform V IAM, KeyCloak.SE (KCSE)         | VM x86 (4/24/150)                        | VM x86 (1/8/150)                       | 2                                         |

Примечания:

- в среде Kubernetes/Platform V DropApp под требованиями к аппаратному обеспечению (CPU, RAM, диски) понимаются
  запросы – минимально необходимые ресурсы (requests), и лимиты – рекомендованные ресурсы (limit) CPU, memory и
  ephemeral-storage, которые Kubernetes предоставляет конкретному контейнеру;
- при горизонтальном масштабировании продукта для подтверждения соответствия ожидаемым нефункциональным характеристикам,
  рекомендуется предварительно провести нагрузочное тестирование на целевой конфигурации конкретной инсталляции.
