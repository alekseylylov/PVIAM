# Проверка параметризации активного healtcheck до бэка в разрезе ответвлений

## Описание

- `PROXY_HEALTHCHECK_ENABLE` - True/False, опциональный, default False, True - включение активного healthcheck до серверов бэка (проверяются ответвления, в которых есть строка '/rds-healthcheck' в applyJctRequestFilter, или `True` в healthcheck.enable), будет вызов GET /healthcheck(изменение через healthcheck.enable) каждые 10 сек.(изменение через healthcheck.interval), и ожидается в ответе http-код 200\302(изменение через healthcheck.validStatus) для живогоузла (статус доступен по url `http://127.0.0.1:10080/status/healthcheck`).
https://sberworks.ru/bitbucket-ci/projects/IAMPROXY/repos/rds/pull-requests/117/diff#distribution/config/rds-server-active-configuration.schema.json

## Шаги проверки

1. **Действие**:

   Выполнить вход в неймспейс проекта auth-iamproxy, перейти по пути /deployments/auth-iamproxy/environment и проверить опцию PROXY_HEALTHCHECK_ENABLE=True

   **Успешный результат**:

   Опция установлена, значение True. Настройки применены успешно.

2. **Действие**:

   Выполнить выполнить изменение конфигурации тестового ответвления. Если не подключен RDS и конфигурация вычитывается из RDS_START_CONF, то в /deployments/auth-iamproxy/yaml на тестовом ответвлении добавить конфигурацию healthcheck:
   ```
   {.....,
   "healthcheck": {
   "enable": true,
   "path": "test/healthcheck",
   "requestBody": "GET %s HTTP/1.1 Host: %s",
   "interval": 3000,
   "timeout": 600,
   "fall": 2,
   "rise": 1,
   "validStatus": "200, 204, 302"
   },
   ...}
   ```
   Сохранить.

   **Тестовые данные**:

   описание параметров:
    "healthcheck": {
    "type": "object",
    "description": "Параметры Healthcheck до back-end",
    "properties": {
    "enabled": {
    "type": "boolean",
    "description": "Использовать Healthcheck до backend"
    },
    "path": {
    "type": "string",
    "description": "Путь в URL до Healthcheck на backend"
    },
    "interval": {
    "type": "number",
    "minimum": 0,
    "description": "Частота вызова HC в миллисекундах"
    },
    "timeout": {
    "type": "number",
    "minimum": 0,
    "maximum": 60000,
    "description": "Ожидание подключения при вызова HC в миллисекундах"
    },
    "fall": {
    "type": "number",
    "minimum": 0,
    "maximum": 100,
    "description": "Количество неуспешных вызовов, для вывода сервера из группы балансировки"
    },
    "rise": {
    "type": "number",
    "minimum": 0,
    "maximum": 100,
    "description": "Количество успешных вызовов, для возврата сервера в группу балансировки"
    },
    "validStatus": {
    "type": "string",
    "pattern": "^[1-5][0-9][0-9](,[1-5][0-9][0-9])*$",
    "description": "HTTP коды ответов, при которых вызовов считается успешным. Пример: 200,204,301,302,403,401"
    },
    "requestBody": {
    "type": "string",
    "description": "Содержимое raw-запроса к Healthcheck backend (включая первую строку запроса и заголовки, при этом с тексте первая %s будет заменена на URL, вторая %s будет заменена на хост). Пример: GET %s HTTP/1.1\\r\\nHost: %s\\r\\n\\r\\n"
    }
    }

   **Успешный результат**:

   Настройки применены успешно.

3. **Действие**:

   Перейти в лог приложения и проверить записи опроса healthcheck до бэкенда

   **Успешный результат**:

   В логах присутствуют записи опроса healthcheck до бэкенда, пример вывода лога если бэкенд недоступен, либо ответ не соответствует validStatus:
   ```2024/12/04 16:19:48 [error] 372#372: *10 [lua] healthcheck.lua:123: errlog(): healthcheck: bad status code from 240.240.146.27:25121: 503, context: ngx.timer
   ```
   запись о Количестве неуспешных вызовов, для вывода сервера из группы балансировки:
   ```2024/12/04 16:19:51 [warn] 377#377: *100 [lua] healthcheck.lua:119: warn(): healthcheck: peer 240.240.146.27:25121 is turned down after 2 failure(s), context: ngx.timer
   ```
   Частота вызова HC соответствует значению установленном в параметре interval (проверить при включенном режиме DEBUG).
