# Подключение и конфигурирование

## Настройка интеграции с Platform V IAM с использованием проксирования

При интеграции с использованием прокси-сервера, клиентская часть аутентификации по `Open ID Connect (Relay Party)`
выполняется на стороне IAM Proxy (AUTH) продукта Platform V IAM (IAM).

Для осуществления проксирования на конкретное приложение/бэкенд, необходимо на новом или уже существующем прокси
сервиса аутентификации добавить "ответвление". 

При добавлении ответвления, необходимо указать базовый контекст (по базовому контексту определяется на какой бэкенд
будет осуществляться проксирование), необходимость вставки в URL базового контекста, адреса и порты серверов приложений
и прочее.

При первом переходе из браузера на прокси, пользователю необходимо пройти аутентификацию. Проксирование на сервера
приложений, возможно после успешной аутентификации.

После проксирования ко всем запросам в бэкенд добавляются HTTP-заголовки:

- `iv-user` - логин пользователя (обратная совместимость с `webseal`);
- `iv-groups` - группы пользователя (обратная совместимость с `webseal`);
- `iv-remote-address` - используется для передачи IP-адреса клиента, обратившегося к серверу. Может включать в себя
  как IPv4, так и IPv6-адреса;
- `Auth-Svc-User` - логин пользователя;
- `Authorization` - аутентификационный токен `OIDC (id_token)` в формате `Authorization: Bearer id_token`.

Сервер приложений (СП) использует для авторизации Авторизацию (AZGT). При вызове методов
API Авторизации из сервера приложений (для получения `ticket`), необходимо использовать **jwt-token** (`id_token` или
`access_token`), который приходит в HTTP-заголовках запроса от IAM Proxy.

Авторизация (AZGT) предназначен для авторизации доступа пользователей на основе **jwt-token**
получаемого от IAM Proxy. Сервис обеспечивает возможность формирования политик авторизации на основе атрибутов из
**jwt-token** и организует процедуру управления ролевыми моделями доступа сервисов Platform V, атрибутами объектов
и субъектов доступа.

В тестовых целях, на стендах тестирования, текущий `id_token/access_token/refresh_token` доступен для просмотра на
endpoint `/jwt/` proxy (доступен только на стендах тестирования).

Пример url - `https://mycompany-auth-svc-proxy-dev2.mycompany.ru/jwt/`

> Примечание: Рекомендуется включать в документацию по установке приложения информацию по настройке ответвлений и
> других параметров IAM Proxy, если это необходимо.

### Выполнение выхода из системы пользователя (сессии IAM Proxy) на стороне приложения

С целью предотвращения несанкционированного доступа к данным конечного пользователя в защищаемом приложении, в IAM Proxy
реализованы функции деактивации пользовательской сессии (выход из системы) и инвалидации JWT-токена. Данные механизмы
являются частью реализации стандарта `OpenID Connect`, что позволяет использовать их практически с любым провайдером
идентификации, поддерживающими этот стандарт.

> В данном разделе IAM Proxy применяется в сценариях работы конечного пользователя с UI приложений 
> (например проксирование при работе с UI приложения в браузере).
>
> Сценарии межсервисных взаимодействий тут не рассматриваются.

Чтобы выполнить выход пользователя (выход из системы) из защищаемого приложения (стоящего за IAM Proxy), необходимо на
фронтенд приложения (на стороне агента, браузера, мобильного приложения), вызвать метод завершения сессии IAM Proxy,
выполнив вызов по относительному URL `/openid-connect-auth/logout` (как пример, из JS это может быть
`window.location.replace("/openid-connect-auth/logout")`, без JS, используя атрибут `href`
, `<a href="/openid-connect-auth/logout">ВЫХОД</a>`). В результате этого вызова IAM Proxy вернет ответ с
перенаправлением (redirect) на endpoint завершения сессии провайдера аутентификации, который после завершения сессии
также вернет ответ с перенаправлением (redirect) обратно на IAM Proxy (на страницу с сообщением об успешном выходе).

В ходе разработки приложения необходимо учитывать особенности реализации завершения сессии пользователя, накладываемые
спецификацией `OpenID Connect` и реализацией компонента IAM Proxy, а именно учитывать что:

- используется несколько перенаправлений (redirect) подряд (их число может меняться, в зависимости от настроек и
  особенностей реализации провайдера аутентификации);
- перенаправления (redirect) могут быть вне текущего домена (домен/fqdn провайдера аутентификации обычно отличается от
  домена/fqdn на котором размещен IAM Proxy), и нужно учитывать правила/ограничения агента связанные с CORS;
- в ответах (включая перенаправления) могут изменяться cookie (приходит заголовок `Set-Cookie`);
- если у приложения есть своя сессия, и необходимо делать ее завершение, то рекомендуется вначале вызывать URL метода
  завершения сессии приложения, который будет возвращать перенаправление на относительный URL метода завершения
  сессии IAM Proxy.

Рекомендации по реализации:

- вызываемый URL метода завершения сессии пользователя (или `logout`), должен быть настраиваемым в вашем приложении, так
  как конкретные значения могут быть изменены, в зависимости от настроек или версии IAM Proxy;
- при проектировании приложения и реализации завершения сессии пользователя, необходимо учесть, что используемый
  протокол `OpenID` и текущая реализация в IAM Proxy, предполагает работу конечных пользователей (реальных людей,
  использующих браузеры или мобильные устройства) с сервисом IAM Proxy. В связи с этим, рекомендуется избегать
  использования решений предназначенных для вызова API при выполнении запросов завершения сессии IAM Proxy. Примером,
  такого нерекомендуемого решения может являться использование в браузере `AJAX`/`fetch` для вызова URL метода
  завершения сессии, при этом непосредственно браузер не обрабатывает результат вызова (его будет обрабатывать JS, т.е.
  код приложения), и такие вызовы браузером считаются небезопасными из-за возможности `XSS`, вследствие чего будут
  применены политики CORS, что в итоге может не позволить выполнить перенаправления на `IDP` провайдера (потребуется
  разрешение `CORS` на `IDP`). Вызов URL метода завершения пользовательской сессии желательно выполнять UI движком
  браузера (заменой `window.location`, или переход по `href`), это позволяет более просто и безопасно обработать вызовы
  и перенаправления, избежать CORS, и не потерять изменения cookie.

#### Опции метода logout в IAM Proxy

При вызове метода завершения сессии IAM Proxy можно изменить его поведение, заданное по умолчанию, и для этого указать в
URL следующие `GET` параметры:

- `revoke` - возможные значения `true`/`false`. Включение/выключение отзыва (`revoke`) токенов при `logout`, через
  прямой вызов API с IAM Proxy в API IDP;
- `logout_idp` - возможные значения `true`/`false`. Включение/выключение вызова `logout` на `IDP` через
  перенаправление (redirect);
- `backchannel` - возможные значения `true`/`false`, по умолчанию `true`. Включение бэкенд вызов `logout` по текущей
  сессии, то есть по которой был получен refresh-токен (в случае с keycloak, будет завершена только текущая сессия
  пользователя). При значении `True` опция `iamproxy.k8s.oidc.revoke_tokens_on_logout` не применяется, и `revoke` не
  производится. Если оставить пустым, будет использоваться значение по умолчанию.

Пример: `/openid-connect-auth/logout?revoke=true&logout_idp=false`.

При таких параметрах будет выполнен `revoke` токенов, а `logout` на `IDP` выполнен не будет, ограничившись лишь выходом
из клиента/АС, соответствующего отозванным токенам.

Описание опций `logout`, задающие поведение по умолчанию, можно посмотреть в демо-профиле развертывания
(опции `oidc_revoke_tokens_on_logout` и `oidc_disable_logout_in_idp`).

### Настройка дизайна стартовой страницы IAM Proxy

IAM Proxy позволяет использовать свой дизайна стартовой страницы. Для этого необходимо заменить файл стартовой
html-страницы `index.html` на свой.

#### Для виртуальной машины (VM)

##### Замена стартовой страницы на VM с использованием профиля развертывания

1. При расположении файлов стартовой страницы и ресурсов для нее, необходимо в профиле развертывания соблюсти
   следующую структуру каталогов (относительно директории профиля развертывания):

   - расположение стартовой страницы: `files/proxy/html/index.html`;
   - ресурсы используемые стартовой страницей (опционально) размещаются по пути `files/proxy/html/`, например:
     `files/proxy/html/my_logo.png`.

2. Внести изменения в профиль развертывания и запустить `Jenkins Job` для установки IAM Proxy (или запустить
   ansible-playbook вручную).

##### Замена стартовой страницы на VM вручную

Для замены стартовой страницы, выполните следующие действия:

1. Скопируйте на VM персонализированную страницу `index.html` и необходимые ей ресурсы в папку `/opt/iamproxy/html/` (
   если есть ответвление на корневой контекст, то надо использовать папку `/opt/iamproxy/conf/html_custom/`).
2. Измените все для скопированных файлов владельца и группу на `iamproxy`, и права на `0440`:
   ```shell
   chown iamproxy:iamproxy index.html
   chmod 0440 index.html
   ```

#### Для k8s (в частности для OpenShift)

##### Замена шаблона стартовой страницы в контейнере

1. **Подготовка `ConfigMap`.**
   Подготовьте персонализированный файл стартовой страницы `index.html`. Для подключения статических бинарных
   ресурсов, если они используются в `index.html`, необходимо выполнить конвертацию бинарных файлов в формат `base64` (
   например командой `base64 my_logo.png`). Пример конвертированного файла:
   `iVBORw0KGgoAAAANSUhE....AAACXBIWXMAABYlAAAWJQFJUiTwAAAA` (полученные строки используем на следующем шаге).

2. **Создание `ConfigMap`.**
   Создайте манифест `ConfigMap` по следующему шаблону:

   ```yaml
   kind: ConfigMap
   apiVersion: v1
   metadata:
     name: iamproxy-custom-index
   data:
     # Содержимое персонализированного index.html
     index.html: |
       <html>
       ......
       </html>
   binaryData:
     # бинарный файл изображения в формате base64
     my_logo.png: "iVBORw0KGgoAAAANSUhE....AAACXBIWXMAABYlAAAWJQFJUiTwAAAA"
     my_background.png: "iVBORw0KGgo....JUiTwAAAA"
   ```

3. **Подготовка `Deployment` IAM Proxy.**
   Подключите созданный в пункте 2 `ConfigMap` к контейнеру IAM Proxy в качестве
   `volume`, для чего приведите `Deployment` IAM Proxy к следующему виду:

   ```yaml
      #...
      spec:
        template:
          spec:
            volumes:
              - name: iamproxy-custom-index-volume
                configMap:
                  name: iamproxy-custom-index
                  defaultMode: 0400 #Права только на чтение
      #...
            containers:
                volumeMounts:
                  - name: iamproxy-custom-index-volume
                    mountPath: /iamproxy/html/index.html
                    subPath: index.html
                  - name: iamproxy-custom-template-volume
                    mountPath: /iamproxy/html/my_logo.png
                    subPath: my_logo.png
                  - name: iamproxy-custom-template-volume
                    mountPath: /iamproxy/html/my_background.png
                    subPath: my_background.png
      #...
   ```

### Использование CORS с IAM Proxy

Для использования ресурсов IAM Proxy при CORS-запросах, необходимо на ответвлении включить опцию
`rds-enable-cors.location.conf`. Так же, чтобы обеспечить работу cookie (аутентификации) при CORS, необходимо чтобы
вызывающая страница и источник размещались на одном родительском домене. Родительский домен необходимо задать как домен
для сессионных cookie IAM Proxy (параметр `iamproxy.k8s.session_domain` при использовании DOT), и для сервиса задать
уникальное имя сессионной cookie (параметр `iamproxy.k8s.session_name` при использовании DOT).

При CORS-запросах предполагается, что аутентификация ранее была сделана пользователем на источнике ресурсов (или для нее
не требуется действий от пользователя), и запросы через IAM Proxy будут проходить успешно. Если действующей
сессии/аутентификации нет, то будет получен ответ с кодом 401 или перенаправление на аутентификацию в провайдере.

При использовании CORS, также необходимо обеспечить отправку cookie в CORS-запросах. Для вызовов из `JS`, это
`XMLHttpRequest` с `withCredentials="true"` или `fetch` с `credentials='include'`, или аналогичные опции для
используемого фреймворка. Для тегов `img`, `audio`, `link`, `script`, `video` это атрибут
`crossorigin="use-credentials"`.

> При необходимости аутентификации в `iframe`, где потребуются действия от пользователя, от провайдера может
> потребоваться отдавать заголовок `Content-Security-Policy: frame-ancestors 'self' https://mycompany.ru`,
> чтобы разрешить отображение контента от провайдера в UI.

> По умолчанию доступны только заголовки ответов, разрешенные стандартом CORS. Для получения доступа к другим
> заголовкам, необходимо указать их в параметре `cors_expose_headers`. Подробное описание параметра приведено в разделе
> [Соответствие имен параметров для разных сред и инструментов установки](../installation-guide/params.md).

### Рекомендации по разработке UI-части систем

При работе в браузере или других агентах имеются определенные ограничения в части передачи cookie, которые следует
учитывать при разработке. Так как для прохождения запросов через IAM Proxy требуется наличие сессионной cookie IAM
Proxy, то необходимо обеспечить наличие сессионной cookie в запросах (по умолчанию это все cookie с префиксом
`PLATFORM_SESSION*`).

Например, при выполнении запроса к manifest.json (при использовании `<link rel="manifest"/>`), необходимо в теге `link`
указать атрибут
`crossorigin` со значением `use-credentials`, чтобы браузер к запросу приложил cookie, даже если при запросе не
возникает CORS.

Например:
`<link rel="manifest" href="/app.webmanifest" crossorigin="use-credentials" />`.
