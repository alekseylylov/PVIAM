## Доступность

- **SLA**: 99.9% времени работы в промышленной среде.
- **Механизмы отказоустойчивости**:
    - горизонтальное масштабирование в Kubernetes;
    - балансировка нагрузки через NGINX Ingress;
    - активный healthcheck компонентов.
  
## Устанавливаемость

- **Поддерживаемые платформы**:
    - linux (Alt Linux, Red Hat Enterprise Linux, SberLinux OS Server);
    - kubernetes (1.24+), OpenShift;
    - docker Engine (20.10+).

- **Инструменты развертывания**:
    - кластеры для автоматической установки в Kubernetes;
    - cli-скрипты для ручного развертывания;
    - документация по установке: [Установка](../installation-guide/installation.md).

- **Совместимость версий**:
    - обеспечение обратной совместимости между версиями IAM Proxy;
    - поддержка обновлений без перезапуска сервисов (Hot Update).

## Целостность

- **Защита данных от модификации**:
    - использование хэширования (SHA-256) для контроля целостности конфигурационных файлов;
    - цифровые подписи для проверки подлинности обновлений;
    - интеграция с системой аудита для фиксации изменений.

- **Контроль целостности при запуске**:
    - проверка целостности образов контейнеров перед запуском;
    - автоматическое обнаружение и блокировка поврежденных компонентов.

- **Защита от коррупции данных**:
    - резервное копирование с проверкой целостности (RAID + CRC32);
    - логирование всех изменений конфигураций с уникальными идентификаторами.

## Надежность

- **Механизмы восстановления**:
    - автоматическое восстановление после сбоев через Kubernetes Self-Healing;
    - регулярные тесты отказоустойчивости с имитацией сбоя узлов;
    - резервное копирование данных с возможностью восстановления до любого момента времени (Point-in-Time Recovery).

- **Стабильность работы**:
    - тестирование на нагрузку с использованием Apache JMeter и Locust;
    - мониторинг метрик через Prometheus и Grafana для выявления аномалий;
    - автоматическая отправка alert при превышении пороговых значений.
  
## Устойчивость

- **Отказоустойчивость**: Failover между провайдерами аутентификации.
- **Резервирование**: Редкий ротационный бэкап конфигураций.
- **Мониторинг**: Prometheus + Grafana для метрик и alert.

## Масштабируемость

- **Горизонтальное масштабирование**: Репликация Pod'ов в Kubernetes.
- **Ограничения на ответвления**:
    - в DropApp: максимум 321 ответвление (при среднем размере 400 символов);
    - в Openshift: 1500 ответвлений (лимит 1048576 байт в ConfigMap).
- **Динамическое расширение**: Автоматическое увеличение ресурсов при высоких нагрузках.

## Ограничения количества символов для определения ответвлений

Допустимое количество ответвлений в средах контейнеризации - не более 321 (при среднем размере одного ответвления в 400 символов).
Ограничение связанно с размером содержимого env-параметров в Deployment, и их передачей в контейнер,
и поэтому точное количество зависит от фактического размера в байтах строки, содержащей конфигурацию ответвлений.

При определении количества более 321 появляется ошибка:
- в DropApp - `exec /entrypoint/entrypoint.sh: argument list too long`;
- в Openshift - `standard_init_linux.go:219: exec user process caused: argument list too long`.

В случае определения ответвлений через configmap предел:
- в DropApp - 1000 ответвлений, значения выше вызывают отказ работы кнопки `save` в веб-интерфейсе;
- в Openshift - 1500 ответвлений, значения выше вызывают ошибку создания объекта `configmap Error "Too long: must
  have at most 1048576 bytes" for field "[]"`.

> Примечание:
> Данные значения по количеству ответвлений зависят в первую очередь от размера данных передаваемых в контейнер,
> которые напрямую зависят от количества символов.
> Также, тесты проводились при конфигурации IAM Proxy без использования RDS Server.
>
> При размере 1048576 байт срабатывает ограничение среды по размеру (в байтах).
> В части функциональности IAM Proxy, на используемых средах развертывания (Openshift, DropApp, VM SberLinux),
> ограничений по количеству ответвлений выявлено не было.
