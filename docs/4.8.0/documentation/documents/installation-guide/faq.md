# Часто встречающиеся проблемы и пути их устранения

Данный раздел содержит типовые проблемы, возникающие при установке и развертывании IAM Proxy, и рекомендации по их устранению.

## Ошибка 1: Время жизни сессии отличается на разных стендах (ПСИ, ИФТ и др.)

### Проблема
На разных стендах время жизни сессии пользователя в IAM Proxy различается, что приводит к неожиданному завершению сессии.

### Причина
Различие вызвано разными значениями параметра `proxy_session_idletime` в профилях установки на разных стендах, а также различным временем жизни access/refresh токенов на стороне IDP.

### Решение
1. Убедитесь, что параметр `proxy_session_idletime` одинаков во всех профилях установки:
   ```ini
   proxy_session_idletime = 3600
   ```
2. Проверьте настройки IDP (например, Keycloak, Okta): убедитесь, что время жизни refresh-токена согласовано с политикой сессии.
3. При использовании Vault для хранения конфигурации — проверьте, что секрет `proxy_session_idletime` одинаков на всех стендах.
4. Перезапустите IAM Proxy после изменения конфигурации:
   ```shell
   systemctl restart iamproxy
   ```

## Ошибка 2: Многократные запросы к IDP при обновлении токенов

### Проблема
IAM Proxy отправляет большое количество запросов к IDP при попытке обновить токены, что вызывает ошибки и нагрузку на IDP.

### Причина
Приложение-клиент не корректно обрабатывает HTTP 401 от IAM Proxy и повторно отправляет множество запросов, каждый из которых инициирует попытку обновления токена.

### Решение
1. Обновите IAM Proxy до версии **Platform V IAM 1.8.0 или выше**, где реализована защита от повторных запросов к IDP.
2. Настройте клиентское приложение на корректную обработку 401:
    - При получении 401 — прекратить автоматические повторы;
    - Перенаправить пользователя на страницу входа.
3. Проверьте логи IAM Proxy:
   ```shell
   journalctl -u iamproxy | grep "token refresh failed"
   ```
4. При необходимости ограничьте частоту запросов на уровне API Gateway или Istio.

## Ошибка 3: Необходимо передавать оригинальные порты через IAM Proxy при использовании Istio

### Проблема
При использовании Istio требуется сохранять оригинальные порты бэкендов, но IAM Proxy их изменяет.

### Причина
IAM Proxy автоматически использует внутренние порты Service Mesh (Istio/SSM) для маршрутизации. Это поведение заложено в механизме интеграции с сервисной сетью.

### Решение
1. IAM Proxy **не поддерживает отключение замены портов** — это техническое ограничение.
2. Для получения реального порта используйте заголовки, добавляемые Istio:
   ```nginx
   set $real_port $http_x_forwarded_port;
   ```
3. Либо вычислите порт по логике шаблонов IAM Proxy:
    - Номер порта = базовый порт + номер ответвления.
    - Шаблоны находятся в дистрибутиве: `auth-cfg/package/k8s/base/istio/config/`.
4. При необходимости реализуйте кастомный фильтр через `custom.d/*.conf`.

## Ошибка 4: Невозможно настроить healthcheck с другим endpoint

### Проблема
Требуется использовать healthcheck на endpoint, отличный от `/healthcheck`, но IAM Proxy всегда обращается к `/healthcheck`.

### Причина
До версии Platform V IAM 2.2.0 возможность изменения healthcheck endpoint отсутствовала.

### Решение
1. Обновите IAM Proxy до версии **2.2.0 или выше**.
2. В конфигурации ответвления задайте:
   ```ini
   proxy_healthcheck_enable = true
   proxy_healthcheck_path = /custom-health
   ```
3. Убедитесь, что бэкенд отвечает на указанный путь с кодом 200, 204 или 302.
4. В более ранних версиях — настройте редирект `/healthcheck → /custom-health` на уровне бэкенда.

## Ошибка 5: Невозможно указать несколько ролей в authorizeByRoleTemplate

### Проблема
Требуется проверить наличие одной из нескольких ролей, но параметр `authorizeByRoleTemplate` принимает только одно значение.

### Причина
В версиях ниже 2.1.0 используется Lua-регулярка без поддержки PCRE.

### Решение
1. Обновите IAM Proxy до версии **2.1.0 или выше**.
2. Используйте формат PCRE:
   ```ini
   authorizeByRoleTemplate = regex:(admin|moderator|user-.*)
   ```
3. В старых версиях — используйте Lua-синтаксис:
   ```ini
   authorizeByRoleTemplate = admin or moderator or user%-.*
   ```
   (обратите внимание: в Lua символ `.` экранируется как `%.`, `*` — как `*`, `|` недоступен).

## Ошибка 6: "Connection timed out" при подключении к бэкенду

### Проблема
IAM Proxy выдает ошибку `upstream timed out` при подключении к бэкенду.

### Причина
Превышен тайм-аут подключения на уровне TCP (по умолчанию — 3 секунды). Бэкенд не отвечает вовремя.

### Решение
1. Проверьте доступность бэкенда:
   ```shell
   telnet backend-host 8080
   ```
2. Увеличьте тайм-аут через кастомную конфигурацию:
   Создайте файл `/opt/iamproxy/conf/custom.d/timeout.location.conf`:
   ```nginx
   proxy_connect_timeout 10s;
   ```
3. Добавьте в профиле установки для нужного ответвления:
   ```yaml
   applyJctRequestFilter: custom.d/timeout.location.conf
   ```
4. Перезапустите IAM Proxy:
   ```shell
   systemctl restart iamproxy
   ```

## Ошибка 7: Невозможно диагностировать медленный бэкенд

### Проблема
Неясно, является ли бэкенд причиной тайм-аутов.

### Причина
Отсутствие инструментов для нагрузочного тестирования вне IAM Proxy.

### Решение
1. Запустите диагностический скрипт с хоста IAM Proxy:
   ```shell
   #!/bin/bash
   count=50
   url=https://backend-host:8080/healthcheck
   outputPath=diag_$(date +%F_%H-%M)
   mkdir -p $outputPath
   for i in $(seq 1 $count); do
     curl -w '%{time_total}' -o /dev/null -s "$url" > $outputPath/result-$i.txt
   done
   ```
2. Проанализируйте результаты:
   ```shell
   awk '{print $1}' diag_*/result-*.txt | sort -n
   ```
3. Проведите тест в периоды низкой и высокой нагрузки.
4. При необходимости — настройте мониторинг производительности бэкенда.

## Ошибка 8: Пользователь получает 401 вместо 302 редиректа

### Проблема
При отсутствии сессии пользователь видит 401, а не перенаправление на IDP.

### Причина
IAM Proxy определяет тип запроса по заголовку `Accept`. Если он не содержит `text/html`, запрос считается API.

### Решение
1. Убедитесь, что браузер отправляет:
   ```
   Accept: text/html,application/xhtml+xml
   ```
2. Настройте параметры распознавания UI:
   ```ini
   proxy_request_to_ui_regex = .*text/html.*|.*application/xhtml.*
   ```
3. Либо принудительно направляйте UI-запросы через кастомный фильтр:
   ```nginx
   if ($http_user_agent ~* Chrome|Firefox) {
       set $redirect_to_login 1;
   }
   ```

## Ошибка 9: "unknown 'o' variable" при старте IAM Proxy

### Проблема
IAM Proxy не запускается, ошибка: `unknown "o" variable`.

### Причина
В значении параметра (например, `proxy_session_secret`) используется строка с символом `$`, который интерпретируется как переменная в nginx.

### Решение
1. Проверьте значение `proxy_session_secret` в профиле или Vault:
   ```ini
   proxy_session_secret = my$s3cr3t$value
   ```
2. Экранируйте `$` как `\$` или удалите его:
   ```ini
   proxy_session_secret = my\$s3cr3t\$value
   ```
3. В версиях 2.1.0+ можно использовать любые символы — проблема устранена.

## Ошибка 10: Неизвестный формат access-log

### Проблема
Невозможно интерпретировать логи запросов IAM Proxy.

### Причина
Формат логов не задокументирован в текущем окружении.

### Решение
1. См. раздел [События системного журнала](../administration-guide/system-log.md).
2. Пример строки:
   ```
   192.x.x.1 - - [01/Jan/2023:12:00:00 +0000] "GET /api/users HTTP/1.1" 200 1234 "-" "curl/7.68.0" "session123" "user@company.ru"
   ```
3. Поля:
    - IP, время, метод, путь, статус, размер, User-Agent, сессия, email.

## Ошибка 11: Неясно, как IAM Proxy определяет аутентификацию

### Проблема
Не понятно, по каким данным IAM Proxy считает пользователя аутентифицированным.

### Причина
Механизм сессий не описан в контексте установки.

### Решение
1. Аутентификация определяется по наличию cookie `PLATFORM_SESSION*`.
2. Проверьте:
   ```shell
   curl -v --cookie "PLATFORM_SESSION=valid_token" https://iam-proxy.local/api
   ```
3. Префикс можно изменить:
   ```ini
   proxy_session_name = CUSTOM_SESSION
   ```

## Ошибка 12: Healthcheck не проходит

### Проблема
Бэкенд помечен как "недоступный", хотя работает.

### Причина
Endpoint `/healthcheck` недоступен или возвращает не 200/204/302.

### Решение
1. Убедитесь, что бэкенд отвечает на `/healthcheck`.
2. Проверьте сетевую доступность:
   ```shell
   kubectl exec -it <iamproxy-pod> -- curl -k http://backend-svc:8080/healthcheck
   ```
3. В SSM укажите явные IP в `ServiceEntry`, если используется DNS resolve.

## Ошибка 13: Не работает отказоустойчивость без healthcheck

### Проблема
При падении бэкенда трафик продолжает на него идти.

### Причина
Активный healthcheck отключен, используется только passive-проверка.

### Решение
1. Включите активный healthcheck (подробнее в разделе [Ошибка 4](#ошибка-4-невозможно-настроить-healthcheck-с-другим-endpoint)).
2. Или настройте passive-проверку:
   ```nginx
   max_fails = 1
   fail_timeout = 10s
   ```
3. По умолчанию nginx пробует следующий сервер при тайм-ауте или ошибке соединения.

## Ошибка 14: IAM Proxy не соответствует требованиям КБ

### Проблема
Система не проходит аудит по требованиям информационной безопасности.

### Причина
Некорректные настройки TLS, OCSP, прав доступа.

### Решение
1. TLS:
   ```nginx
   ssl_protocols TLSv1.2 TLSv1.3;
   ```
2. OCSP:
   ```nginx
   ssl_stapling on;
   ssl_stapling_verify on;
   ```
3. Проверка сертификатов:
   ```nginx
   proxy_ssl_verify on;
   ```
4. Запуск от непривилегированного пользователя:
   ```ini
   [Service]
   User=iamproxy
   Group=iamproxy
   ```
5. Все пути в `include` — относительные.

## Ошибка 15: SSL-ошибка при подключении к бэкенду

### Проблема
IAM Proxy не может установить TLS-соединение с бэкендом.

### Причина
Сертификат бэкенда не доверен или просрочен.

### Решение
1. Проверьте срок действия сертификата:
   ```shell
   echo | openssl s_client -connect backend:8443 2>/dev/null | openssl x509 -noout -dates
   ```
2. Убедитесь, что CA добавлен в `trusted_chain.crt.pem`.
3. Укажите путь в конфиге:
   ```nginx
   proxy_ssl_trusted_certificate /opt/iamproxy/conf/certs/trusted_chain.crt.pem;
   ```
4. Перезагрузите конфигурацию:
   ```shell
   systemctl reload iamproxy
   ```
