# Механизмы безопасности

Обеспечиваются следующие механизмы безопасности:

- регистрация событий аудита, связанных с аутентификацией пользователей;
- изолированное хранение секретов с использованием SecMan;
- аутентификация осуществляется с помощью OpenID Connect с authorization code flow, для которого Relying Party является
  IAM Proxy;
- для входящих взаимодействий (от пользователя к IAM Proxy) используется TLS (версии 1.2 и 1.3), при необходимости можно
  включить аутентификацию по клиентскому сертификату с проверкой его DNS-кода по заданному регулярному выражению;
- контроль аутентификации осуществляется по наличию в HTTP-запросах действующей сессии IAM Proxy (по умолчанию это
  cookie с именами `PLATFORM_SESSION*`), в следствии чего, через IAM Proxy должен проходить весь трафик от
  браузера/агента в сторону приложения/бэкенд;
- IAM Proxy позволяет ограничить время жизни сессии по неактивности, по времени действия токена от OIDC-провайдера,
  выполнить привязку по клиентскому IP;
- ограничение обращения к серверам приложений на основе обращения к внешнему серверу авторизации, что позволяет вынести
  функции безопасности на отдельный компонент (прокси) и упростить авторизацию на приложении.

## Примечание

Подробнее в разделе [Руководство по безопасности](../security-guide/index.md).

## Идентификация

IAM Proxy не реализует собственные механизмы идентификации — он делегирует ее внешним системам:
- для пользователей — через OIDC/SAML/LDAP;
- для сервисов — через mTLS или Service Mesh.

## Аутентификация

Аутентификация реализуется через **OpenID Connect (OIDC)** с использованием `authorization code flow`, где IAM Proxy выступает в роли Relying Party. Для входящих запросов от пользователей применяется **TLS 1.2/1.3**, а при необходимости — аутентификация по клиентскому сертификату с проверкой его DNS-кода через регулярное выражение. 

Контроль аутентификации осуществляется через наличие действующей сессии IAM Proxy, которая хранится в HTTP-заголовках (`cookie` с именами `PLATFORM_SESSION*`). Сессии могут быть ограниченны по времени жизни: 
- по неактивности;
- по времени действия токена OIDC-провайдера;
- с привязкой к IP-адресу клиента.

## Авторизация

Авторизация выполняется через IAM Proxy, который ограничивает доступ к серверам приложений на основе правил, заданных внешним сервером авторизации. Это позволяет вынести логику безопасности на уровень прокси, упрощая реализацию авторизации в самом приложении. 

## Аутентификация и авторизация

Процессы аутентификации и авторизации интегрированы в IAM Proxy, который выступает как центральная точка контроля. После успешной аутентификации через OIDC, IAM Proxy проверяет права доступа пользователя и фильтрует запросы согласно политикам, заданным в системе авторизации.

## Разграничение доступа

Разграничение доступа реализуется через политики, определяемые в IAM Proxy. Эти политики могут включать:
- ограничение доступа по группам пользователей;
- географическим ограничениям;
- временным рамкам (например, доступ только в рабочее время);
- привязку к IP-адресам или диапазонам.

## Обеспечение целостности данных

Целостность данных обеспечивается за счет использования **TLS 1.2/1.3**, который включает механизмы проверки целостности передаваемых данных. Дополнительно, при использовании клиентских сертификатов, их DNS-код проверяется через регулярные выражения, исключающие подделку идентичности клиента.

## Защита данных

Для защиты данных применяются следующие меры:
- **Изолированное хранение секретов** с использованием SecMan, что предотвращает их утечку.
- **Шифрование данных в покое и в движении** через TLS.
- **Клиентская аутентификация** по сертификатам для чувствительных операций.

## Шифрование

Все коммуникации между клиентом и IAM Proxy, а также между IAM Proxy и внутренними сервисами, защищены **TLS 1.2/1.3**. Для дополнительной безопасности поддерживаются:
- клиентские сертификаты с проверкой DNS-кода;
- шифрование секретов в SecMan.

## Аудит

Реализуется регистрация событий аудита, связанных с аутентификацией пользователей. Все попытки входа, ошибки аутентификации и изменения в сессиях фиксируются для последующего анализа. Подробнее см. [Руководство по безопасности](../security-guide/index.md).
