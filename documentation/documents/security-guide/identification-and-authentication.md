# Идентификация и аутентификация

## Идентификация и аутентификация пользователей

### Варианты идентификации пользователей
- **Идентификация через провайдера OpenID Connect (OIDC)**:  
  IAM Proxy не хранит учетные данные пользователей. Идентификация осуществляется через внешний провайдер (например, Platform V IAM KeyCloak.SE или другие OIDC-совместимые системы).
- **Единая точка входа (SSO)**:  
  Поддержка однократной аутентификации для доступа к нескольким системам через один провайдер.

### Используемые идентификаторы пользователей

| Наименование идентификатора  | Формат                           | Уникальность |
|------------------------------|----------------------------------|--------------|
| Логин                        | Строка (латинские буквы, цифры)  | Уникален     |
| Email                        | Стандартный email                | Уникален     |
| Subject (sub) из токена      | UUID                             | Уникален     |

> **Примечание**: Идентификаторы генерируются и управляются провайдером аутентификации (OIDC).

### Настройка идентификаторов пользователей
- **Управление через провайдер**:  
  Создание, изменение и удаление пользователей выполняется в консоли управления провайдера (например, KeyCloak.SE).
- **Синхронизация с внешними системами**:  
  Интеграция с любым OIDC-провайдером.

### Варианты аутентификации пользователей
1. **Аутентификация по логину и паролю**:  
   Провайдер проверяет учетные данные и возвращает токены после успешной верификации.
2. **Многофакторная аутентификация (MFA)**:
   - OTP (Яндекс Ключ, Google Authenticator, FreeOTP);
   - аппаратные ключи (YubiKey).
3. **Аутентификация через SAML/LDAP**:  
   Через интеграцию с провайдерами, поддерживающими эти протоколы.

### Используемые аутентификаторы пользователей
- **Пароль**:  
  Установите минимальную длину пароля не менее 12 символов (буквы верхнего и нижнего регистра, цифры, специальные символы).
- **Токены TOTP**:  
  Генерируются приложениями вроде Google Authenticator.
- **Цифровые сертификаты**:  
  Используются для MFA в корпоративных средах.

## Настройки параметров аутентификации пользователей
- **Срок жизни токенов**:
   - `access_token`: 1 час (настраивается на IDP);
   - `refresh_token`: 24 часа (настраивается на IDP).
- **Обновление токенов**:  
  Автоматическое обновление через `refresh_token` без участия пользователя.
- **Проверка токенов**:
   - интервал проверки актуальности токена: `oidc_introspection_interval` (настраивается в конфигурации IAM Proxy);
   - ручная проверка через `/introspect` API провайдера.

### Правила задания и смены паролей
**Правила задания**:
- **Максимальный срок действия пароля**: 80 дней (форсированная смена).
- **Минимальная длина**: **12 символов**.
- **Обязательные элементы**:
  - **1 буква в нижнем регистре** (например, `a`, `b`);
  - **1 буква в верхнем регистре** (например, `A`, `B`);
  - **1 цифра** (например, `0–9`);
  - **1 спецсимвол** (`! " # $ % &` и т.д.).
- **Пароль должен содержать минимум 6 уникальных символов** (независимо от длины).
- **Запрещены последовательности из 3 и более рядом расположенных символов на клавиатуре** (например, `QWE`, `123`, `ASD`).
- **Пароль не должен совпадать с логином пользователя**.
- **Запрет на повторение 24 последних паролей**.
- **Алгоритм хеширования**: `pbkdf2-sha512` с **12345 итерациями**.

- **Смена пароля**:  
  Выполняется через интерфейс провайдера (например, KeyCloak.SE).
  - **Форсированная смена**: Администратор может требовать смену пароля при истечении срока действия (**каждые 80 дней**).
  - **Процесс смены**:
    1. Введите **текущий пароль** в поле "Текущий пароль".
    2. Введите **новый пароль**, соответствующий парольной политике, в поле "Новый пароль".
    3. Подтвердите новый пароль в поле "Подтверждение пароля".
    4. Нажмите кнопку **«Сохранить»** для применения изменений.

## Идентификация и аутентификация сервисов

### Варианты идентификации сервисов
- **mTLS (Mutual TLS)**:  
  Использование двусторонней аутентификации по сертификатам между сервисами.
- **Service Mesh (Platform V Synapse Service Mesh)**:  
  Для контейнеризированных сред — аутентификация через SSM.

### Используемые идентификаторы сервисов

| Наименование идентификатора   | Формат                           | Уникальность  |
|-------------------------------|----------------------------------|---------------|
| Common Name (CN) сертификата  | Строка (например, service-name)  | Уникален      |
| Имя сервиса в SSM             | Строка                           | Уникален      |

### Настройка идентификаторов сервисов
- **Генерация сертификатов**:  
  Сертификаты создаются через PKI CA или другую доверенную систему.
- **Интеграция с SSM**:  
  Регистрация сервисов в Service Mesh с указанием их идентификаторов.

### Варианты аутентификации сервисов
1. **mTLS 1.2/1.3**:  
   Использование клиентских и серверных сертификатов для взаимной аутентификации.
2. **Service Mesh (SSM)**:  
   Аутентификация через Sidecar-прокси, управляемые SSM.

### Используемые аутентификаторы сервисов
- **X.509 сертификаты**:  
  Используются для mTLS.
- **Service Mesh Identity**:  
  Уникальный идентификатор сервиса в SSM.

### Настройки параметров аутентификации сервисов
- **Параметры mTLS**:
   - `ssl_protocols`: TLSv1.2, TLSv1.3;
   - `ssl_ciphers`: Высокозащищенные алгоритмы (например, ECDHE-RSA-AES256-GCM-SHA384).
- **Проверка сертификатов**:
   - отключение небезопасных алгоритмов (например, SHA-1);
   - периодическая ротация сертификатов (не реже чем каждые 6 месяцев).
- **SSM Configuration**:
   - настройка доверенных корневых сертификатов в SSM;
   - мониторинг состояния соединений через Prometheus + Grafana.
