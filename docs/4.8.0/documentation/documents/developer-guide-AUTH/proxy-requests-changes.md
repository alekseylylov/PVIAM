# Изменение запросов на стороне IAM Proxy

Во время обработки запросов и ответов, IAM Proxy может изменять/добавлять/удалять определенные HTTP-заголовки (в
зависимости от заданных опций).

## Добавляемые или изменяемые HTTP-заголовки на IAM Proxy

### Перечень добавляемых или изменяемых HTTP-заголовков в запросах к бэкенд

- `X-Real-IP`, IP адрес агента/браузера;
- `X-Forwarded-For`, цепочка из инициаторов соединения через запятую (первый агент/браузер);
- `X-Forwarded-Server`, IP адрес сервера IAM Proxy;
- `X-Forwarded-Proto`, оригинальная схема запроса (http, https), которая использовалась в URL при запросе на
  агенте/браузере;
- `X-Forwarded-Host`, оригинальное имя сервера, которое использовалось в URL при запросе на агенте/браузере;
- `X-Forwarded-Port`, оригинальный порт сервера, который использовался в URL при запросе на агенте/браузере;
- `Connection`, изменяется на `close` для всех запросов, кроме тех, которые содержат HTTP-заголовок `Upgrade`(в целях
  поддержки проксирования WebSocket);
- `Cookie`, содержит cookie запроса, удаляются cookie со следующими именами:
  - начинающиеся на `PLATFORM_SESSION`(этот префикс может быть изменен опцией `proxy_session_name`);
  - сессия `IAM Proxy` (например `PLATFORM_SESSION`, `PLATFORM_SESSION_2`, `PLATFORM_SESSION_ID`);
- `PD-ID`, failover токен ISAM;
- `X-IAM-RQID`, уникальный идентификатор запроса.

### Перечень добавляемых или изменяемых HTTP-заголовков в ответах от бэкенд

В целях диагностики проблем или повышения уровня безопасности, добавляются/заменяются в **ответах от бэкенд** HTTP
заголовки:

- `Server`, идентификатор IAM Proxy;
- `X-Proxy-Request-ID`, уникальный идентификатор запроса;
- `X-Proxy-Upstream`, имя группы серверов (upstream) бэкенд, от которой был получен ответ;
- `X-Proxy` (есть только на стендах типа `nt`,`dev`,`ift`), адрес или имя текущего сервера IAM Proxy;
- `X-Proxy-BackEnd` (есть только на стендах типа `nt`,`dev`,`ift`), IP адрес сервера бэкенд (если было несколько
  попыток, то адреса будут перечисленны через запятую), от которого был получен ответ;
- `X-Proxy-BackEnd-Status` (есть только на стендах типа `nt`,`dev`,`ift`), HTTP-статус/код ответа, который был получен с
  бэкенд (если было несколько попыток, то коды будут перечисленны через запятую);
- `X-Proxy-BackEnd-ConnTime` (есть только на стендах типа `nt`,`dev`,`ift`), время затраченное на подключение к серверу
  бэкенд (если было несколько попыток, то значения будут перечисленны через запятую);
- `X-Proxy-BackEnd-RespTime` (есть только на стендах типа `nt`,`dev`,`ift`), время затраченное на получение ответа от
  сервера бэкенд (если было несколько попыток, то значения будут перечисленны через запятую);
- `Strict-Transport-Security`, устанавливается в `Strict-Transport-Security: max-age=31536000; includeSubDomains`,
  задается запрет в браузере для URL IAM Proxy смены https на http;
- `X-Content-Type-Options`, устанавливается в `nosniff`, включение в браузере проверки соответствия типа MIME (что для
  типов 'script' и 'style', заголовок Content-Type содержит соответствующие значения);
- `Content-Security-Policy`, устанавливается в
  `default-src 'self'; font-src *; img-src * data:; script-src *; style-src *`, задаем для браузера CSP
  (значение можно изменить опцией `cors_header_content_security_policy`).

Для поддержки функциональности CORS (при включенной опции `common/rds-enable-cors.location.conf`),
добавляются/заменяются в **ответах от бэкенд** HTTP-заголовки:

- `Access-Control-Allow-Origin`, разрешенный источник для запроса (фильтр устанавливается опцией `cors_allow_origins`);
- `Vary`, устанавливается в `Origin`, указание заголовка, на основе которого можно кешировать ответ на агенте/браузере;
- `Access-Control-Allow-Credentials`, устанавливается в `true`, разрешаем передавать/принимать cookie в CORS запросах;
- `Access-Control-Allow-Methods` (только для запросов с методом `OPTIONS`), устанавливается в
  `GET, POST, PUT, DELETE, HEAD`, разрешаем использовать указанные HTTP методы для CORS запросов (устанавливается опцией
  `cors_allow_methods`);
- `Access-Control-Allow-Headers` (только для запросов с методом `OPTIONS`), устанавливается в `Origin, Content-Type`,
  разрешаем использовать указанные HTTP-заголовки для CORS запросов (устанавливается опцией `cors_allow_headers`);
- `Access-Control-Max-Age` (только для запросов с методом `OPTIONS`), устанавливается в `600`, разрешаем использовать
  кешированный ответ в течение 10 минут.
- `Access-Control-Expose-Headers`, указывает HTTP заголовки ответа, к которым можно получить доступ из js в браузере (
  задается опцией `cors_expose_headers`);

> По умолчанию из js в браузере доступны только заголовки ответов, разрешенные стандартом CORS. Для получения доступа к другим
> заголовкам, необходимо указать их в заголовке HTTP ответа `Access-Control-Allow-Headers` 
> (задается через параметр `cors_expose_headers`).
> Подробное описание параметра приведено в разделе
> [Соответствие имен параметров для разных сред и инструментов установки](../installation-guide/params.md).

### HTTP-заголовки передаваемые при обращении к провайдеру идентификации

Информация о версиях продукта и Операционной системе, передается в заголовке `Server` в следующем формате:

- для тестовых стендов:

  "PlatformV IAM/2.0.0 AUTH/4.5.3 (IAM Proxy; distribution 4.5.3-123) SberLinux/8.7 (VM; 5 vCPU)"

  Детализация значений заголовка:
  - PlatformV IAM/2.0.0 - код продукта/версия продукта;
  - AUTH/4.5.3 - код компонента/версия компонента;
  - IAM Proxy; distribution 4.5.3-123 - имя компонента; версия установленного дистрибутива;
  - SberLinux/8.7 - наименование ОС/версия ОС;
  - VM; 5 vCPU - тип среды развертывания; количество ядер процессора в используемой инсталляции

  > В случае возникновения проблем в определении типа среды или количества ядер, по умолчанию будут использованы
  > следующие значения:
  > - Unknown_VM - не определена среда развертывания;
  > - 0 vCPU - не определено количество ядер процессора.
  >
  > Пример: "PlatformV IAM/1.8.0 AUTH/4.5.0 (IAM Proxy; 1.8.0-123) SberLinux/8.7 (Unknown_VM; 0 vCPU)"

- для промышленных стендов:

  "PlatformV IAM AUTH (IAM Proxy)"

### Установка Cookie

На любом HTTP-запросе к IAM Proxy в ответ может быть добавлен HTTP-заголовок с установкой cookie
(заголовок `Set-Cookie`), при этом ожидается, что агент его примет и обработает. IAM Proxy для поддержания своего
функционирования устанавливает cookie со следующими именами:

- `PLATFORM_SESSION`, `PLATFORM_SESSION_2`, ... , содержат контекст сессии IAM Proxy (контекст защищается с помощью
  симметричным AES-шифрования на ключе, разделяется блоками по 4000 байт, и каждый блок передается в отдельной cookie с
  префиксом `PLATFORM_SESSION`);
- `PLATFORM_SESSION_ID`, содержит уникальный идентификатор сессии IAM Proxy на агенте (может использоваться для
  идентификации сессии, как на IAM Proxy, так и на узлах до IAM Proxy).

> Примечание:
>
> Префикс `PLATFORM_SESSION` может быть изменен опцией `proxy_session_name`.
>
> Домен на сессионные cookie IAM Proxy по умолчанию не задается
> (но при необходимости может быть задан опцией `proxy_session_domain`),
> `SameSite` устанавливается в `Lax`, срок истечения cookie не задается
> (действует во время текущей сессии агента/браузера), `Secure` задается если схема в браузере HTTPs.
>
> Контекст сессии IAM Proxy может содержать (в зависимости от установленных опций, и состояния сессии) следующие данные:
> - проверочные коды при аутентификации (state, nonce, code_verifier);
> - URL при вызове которого потребовалось пройти аутентификацию;
> - время последней активности пользователя;
> - время аутентификации пользователя;
> - логин пользователя;
> - токены `access_token`, `id_token`, `refresh_token`;
> - атрибуты, полученные от IDP из userinfo;
> - флаг признака действующей аутентификации;
> - флаг признака повторной аутентификации (для однократного повтора при ошибке с state);
> - время, когда истекает `access_token`;
> - время, когда истекает `refresh_token`;
> - hash от `refresh_token`, по которому ранее был отказ на IDP;
> - время, после которого считается корректной реаутентификация.

### Удаляемые HTTP-заголовки на IAM Proxy

В целях повышения уровня безопасности и сохранения конфиденциальности деталей сервера приложений, удаляются в **ответах
от бэкенд** HTTP-заголовки:

- `X-Powered-By`, технология используемая сервером для обработки запросов;
- `X-CF-Powered-By`, платформа для обработки запросов;
- `X-Generator`, инструмент или система, используемая для генерации содержимого;
- `X-AspNet-Version`, версия фреймворк используемая для обработки запросов;
- `X-AspNetMvc-Version`, версия фреймворк используемая для обработки запросов.

В целях повышения уровня безопасности, удаляются в **запросах к бэкенд** HTTP-заголовки:

- `Referer`, содержимое может иметь неожиданные значения, и не рекомендуется использование заголовка для любых целей на
  бэкенд;
- `Origin`, устаревший аналог `Referer`.

> Для определения сервера и схемы запроса, которые использовались в URL на агенте/браузере,
> рекомендуется на бэкенд использовать заголовки `X-Forwarded-Host` и `X-Forwarded-Proto`.

## Изменение содержимого HTTP запросов на IAM Proxy при непрозрачном проксировании

Чтобы обеспечить функциональность непрозрачного проксирования, при котором на IAM Proxy появляется дополнительная часть
в URL (параметр ответвления `junctionPoint`), необходимо изменять URL в HTTP запросах и ответах.

Ниже будут использованы для обозначения изменяемых частей переменные:

- `$jctroot` - значение параметра ответвления `junctionPoint`;
- `$host`, `$http_host` - значение HTTP заголовка `Host` из запроса;
- `$server_jct` - первый адрес сервера бэкенд из настройки ответвления.

### Изменения в HTTP запросе

В HTTP **запросе** изменяется URL (в `HEAD`), из начала которого удаляется `$jctroot`.

Например, запрос при `junctionPoint=/my-jctroot`:

```
GET /my-jctroot/path HTTP/1.0
Content-Type: text/html; charset=utf-8
```

будет изменен на:

```
GET /path HTTP/1.0
Content-Type: text/html; charset=utf-8
```

### Изменения в теле HTTP ответа

В теле HTTP **ответа** производятся замены тогда, когда HTTP заголовок ответа `Content-Type` соответствует одному из
типов:

- `text/html`;
- `text/css`;
- `text/javascript`;
- `application/javascript`;
- `text/json-comment-filtered`;
- `application/manifest+json`;

В теле HTTP **ответа** заменяется:

- `href="/` на `href="$jctroot/`;
- `src="/` на `src="$jctroot/`;
- `src:"/` на `src:"/$jctroot/`;
- `url("/` на `url("/$jctroot/`;
- `"action="/` на `"action="/$jctroot/`;
- `"baseUrl": "/` на `"baseUrl": "/$jctroot/`;
- `window.location.host +"/` на `window.location.host +"$jctroot/`;
- `href="https://$http_host/` на `href="/`;
- `src="https://$http_host/` на `href="/`;
- `"start_url":"https://$http_host/` на `"start_url":"/`.

Тут указаны только базовые варианты замен, фактически есть еще замены модификаций этих вариантов (например с разными
вариантами кавычек, или с пробелами между разделителями и т.п.). Более детальный и актуальный список правил замен можно
посмотреть в конфигурации IAM Proxy, в файле `not-transparent-jct.location.conf`
(в дистрибутиве расположен в `distr/package/bh/iamproxy/config/conf/common/not-transparent-jct.location.conf`).

### Изменения в заголовках HTTP ответа

В HTTP заголовке **ответа** `Location` заменяется (выполняется только первая замена по списку):

- `/openid-connect-auth`                   на `/openid-connect-auth`;
- `https://$http_host/openid-connect-auth` на `/openid-connect-auth`;
- `https://$host/openid-connect-auth`      на `/openid-connect-auth`;
- `https://$host/`       на `$jctroot/`;
- `https://$http_host/`  на `$jctroot/`;
- `/`                    на `$jctroot/`;
- `http://$host/`        на `$jctroot/`;
- `http://$http_host/`   на `$jctroot/`;
- `http://$server_jct/`  на `$jctroot/`;
- `https://$server_jct/` на `$jctroot/`.
