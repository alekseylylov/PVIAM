# Безопасность данных

## Справочная информация

К чувствительной информации, безопасность которой обеспечивается **самим Продуктом**:
- аутентификационные данные (пароли от KeyStore, сертификаты, приватные ключи);
- авторизационные данные (роли, привилегии в контексте Open ID Connect);
- конфигурационные файлы (KeyStore, параметры mTLS);
- журналы аудита (при необходимости).

К чувствительной информации, безопасность которой обеспечивается **средой функционирования**:
- секреты Open ID Connect (`client_id`, `client_secret`);
- сертификаты доверенных ЦС (`trusted_ca_*.cer`, `trusted_ca_*.crt.pem`);
- конфигурационные параметры, хранящиеся в SecMan.

К чувствительной информации, безопасность которой обеспечивается **внешними сервисами**:
- секреты, хранящиеся в SecMan (аналог HashiCorp Vault);
- сертификаты, используемые для HTTPS и mTLS.

## Безопасность данных, обеспечиваемая программным продуктом

### Перечень защищаемой информации
1. **Файлы KeyStore**:
   - `keycloak-keystore.p12` (сертификаты/ключи для Keycloak);
   - `proxy-server.p12` (клиентский сертификат для mTLS);
   - `rds-server-keystore.p12` (сертификаты/ключи для RDS Server).
2. **Приватные ключи**:
   - `proxy-server.key` (ключ сертификата HTTPS).
3. **Сертификаты**:
   - `proxy-server.crt.pem` (сертификат HTTPS).

### Механизмы защиты
- шифрование паролей от KeyStore алгоритмом **AES-256**;
- хранение сертификатов и ключей в защищенных форматах (`.p12`, `.key`, `.crt.pem`);
- ограничение доступа к файлам KeyStore и сертификатам (минимальные привилегии для пользователя `iamproxy`);
- использование Kubernetes/OpenShift для изоляции компонентов IAM Proxy и настройки RBAC.

## Безопасность данных, обеспечиваемая внешними сервисами

### Перечень защищаемой информации
1. **Секреты Open ID Connect**:
   - `proxy_oidc_client_id`, `proxy_oidc_client_secret`, `proxy_oidc_alt_idp_client_secret` и др.
2. **Сертификаты доверенных ЦС**:
   - `trusted_ca_*.cer`, `trusted_ca_*.crt.pem`.

### Механизмы защиты
- централизованное хранение секретов в **SecMan**;
- регулярное обновление ключей и сертификатов;
- интеграция с SecMan для автоматической синхронизации конфигураций.

## Безопасность данных, обеспечиваемая ПО среды функционирования

### Перечень защищаемой информации
- секреты, хранящиеся в SecMan;
- сертификаты доверенных ЦС.

### Механизмы защиты
- шифрование данных в покое (в SecMan);
- использование протоколов **mTLS + HTTPS** для безопасной передачи данных;
- настройка RBAC в Kubernetes/OpenShift для ограничения доступа.

## Места хранения защищаемой информации

| Тип хранилища         | Примеры файлов/ресурсов                          | Описание                                         |
|-----------------------|--------------------------------------------------|--------------------------------------------------|
| Локальные файлы       | `/opt/iamproxy/conf/proxy-server.key`            | Приватные ключи и сертификаты                    |
| KeyStore              | `/opt/iamproxy/keystores/keycloak-keystore.p12`  | Хранилище сертификатов и ключей                  |
| SecMan                | Секреты Open ID Connect                          | Централизованное хранение чувствительных данных  |

## Форма хранения защищаемой информации

| Форма хранения         | Примеры                                         | Описание                                  |
|------------------------|-------------------------------------------------|-------------------------------------------|
| Зашифрованный вид      | Пароли от KeyStore (AES-256)                    | Шифрование конфиденциальных данных        |
| Открытый вид           | Сертификаты (`.crt.pem`)                        | Используются в процессе TLS/HTTPS         |
| Хешированный вид       | Не применяется                                  | Хеширование не используется               |

## Рекомендации по защите данных

1. **Шифрование**:
   - использовать AES-256 для шифрования паролей от KeyStore;
   - хранить секреты в SecMan вместо локальных файлов.

2. **Централизованное управление**:
   - интегрировать SecMan для управления секретами;
   - регулярно обновлять ключи и сертификаты.

3. **Ограничение доступа**:
   - применять принцип least privilege для пользователя `iamproxy`;
   - настроить RBAC в Kubernetes/OpenShift.

4. **Мониторинг и аудит**:
   - анализировать журналы аудита в **Platform V Audit SE**;
   - настроить алерты на подозрительную активность.

## Рекомендации по противодействию потере данных

| Компонент            | Последствия отказа                  | Компенсирующие механизмы                     |
|----------------------|-------------------------------------|----------------------------------------------|
| SecMan               | Недоступность секретов              | Резервное копирование секретов               |
| KeyStore             | Невозможность аутентификации        | Автоматическое резервирование KeyStore       |
| Сертификаты          | Нарушение TLS/HTTPS                 | Регулярное обновление сертификатов           |


## Криптографические методы защиты данных

### Механизмы защиты
- шифрование: AES-256 для паролей от KeyStore;
- подпись JWT-токенов: HMAC-SHA256;
- передача данных: mTLS + HTTPS.

### Криптографические алгоритмы

| Наименование алгоритма  | Длина ключа (бит) | Применение                |
|-------------------------|-------------------|---------------------------|
| AES-256                 | 256               | Шифрование паролей        |
