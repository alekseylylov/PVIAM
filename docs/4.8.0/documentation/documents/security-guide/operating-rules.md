# Правила эксплуатации

Серверная инфраструктура обеспечивает получение реального клиентского IP-адреса. Рекомендуется использовать
proxy-протокол или балансировку без потери реального IP-адреса.

## Методы получения клиентского IP

Методы получения клиентского IP в IAM Proxy:

- src ip – Source Internet Protocol (Исходный Протокол Интернета). Подразумевает прямое использование источника
  IP-адреса, который виден на уровне транспортного протокола (например TCP). Src IP показывает реальный IP-адрес
  клиента, в случаях когда клиент подключен к серверу напрямую;
- XFF – X-Forwarded-For. Используется для идентификации исходного IP-адреса клиента, который подключается к веб-серверу
  через HTTP-прокси или балансировщик нагрузки. Заголовок XFF добавляется прокси-сервером или балансировщиком нагрузки и
  может содержать один или несколько IP-адресов, в случае если клиент прошел через несколько прокси;
- PROXY PROTOCOL – протокол, который передает информацию о клиентском соединении через уровень сетевой инфраструктуры,
  такой как прокси-серверы или балансировщики нагрузки. Протокол добавляет к трафику дополнительную строку с
  метаданными, которая содержит исходный IP-адрес клиента, а также адрес и порт назначения. Дополнительная информация о
  `PROXY PROTOCOL` [по ссылке](https://www.haproxy.org/download/1.8/doc/proxy-protocol.txt).

## Влияние среды функционирования на безопасность

IAM Proxy не имеет собственных программных и программно-аппаратных средств защиты информации.

В качестве внешних средств защиты могут быть использованы:

- аудит (AUDT) продукта Platform V Audit SE (AUD) для ограничения доступа к прикладным логам;
- единый коллектор телеметрии (COTE) продукта Platform V Monitor (OPM) для ограничения доступа к прикладным логам;
- объединенный мониторинг Unimon (MONA) продукта Platform V Monitor (OPM) для ограничения доступа к анализу метрик за
  определенный промежуток времени.

## Правила эксплуатации среды функционирования программного продукта

## 1. Физическая и логическая защита инфраструктуры

### 1.1. Защита серверов

- серверы, на которых установлен IAM Proxy, должны находиться в помещениях с ограниченным физическим доступом (например,
  дата-центры с системой контроля доступа);
- использовать аппаратные средства защиты (токенизаторы, HSM) для хранения критических секретов.

### 1.2. Сетевая безопасность

- настроить фаерволы для ограничения входящего/исходящего трафика:
  - разрешить подключение только через HTTPS (порт 443);
  - запретить доступ к портам управления (например, SSH/22) из внешней сети, разрешая его только через прокси или VLAN.
- использовать IP-фильтрацию для ограничения доступа к API IAM Proxy (например, через параметр `proxy_jct_list`).

## 2. Управление учетными записями и доступом

### 2.1. Политики паролей

- все учетные записи (администраторов, сервисные аккаунты) должны использовать:
  - пароли длиной не менее 12 символов, включающие буквы, цифры и спецсимволы;
  - обязательную смену паролей каждые 80 дней;
  - двухфакторную аутентификацию (2FA) для администраторских аккаунтов.

### 2.2. Минимизация привилегий

- принцип least privilege:
  - администраторы получают минимальный набор прав для выполнения своих задач;
  - клиентские приложения используют сервисные аккаунты с ограниченным доступом.
- отключать учетные записи после завершения работы.

## 3. Защита конфиденциальных данных

### 3.1. Хранение секретов

- все конфиденциальные данные (пароли, токены, сертификаты) должны:
  - храниться в зашифрованном виде (например, в файлах `vault_main_decrypted.yml`);
  - не допускаться к хранению в исходных кодах или незащищенных конфигурациях.
- использовать систему управления секретами (Secrets Manager) для автоматической доставки ключей в runtime.

### 3.2. Шифрование данных

- все коммуникации между компонентами IAM Proxy (Keycloak, Audit Service) должны происходить по защищенным
  каналам (TLS 1.2+);
- настроить шифрование данных на диске (LUKS/FDE) для серверов, где хранятся конфигурационные файлы.

## 4. Логирование и мониторинг

### 4.1. Ведение журналов

- включить детальное логирование всех действий:
  - аудит событий (логины, изменения конфигураций) через `audit2_options`;
  - журнализация ошибок и предупреждений (например, `platformauth.log`).
- хранить логи в защищенной системе (ELK Stack, Splunk), исключая возможность модификации.

### 4.2. Мониторинг и алертинг

- настроить мониторинг метрик (CPU, память, загрузка сети) через Prometheus/Grafana;
- установить алерты на:
  - необычную активность (например, частые ошибки аутентификации);
  - падение доступности сервисов (проверка `oidc_idp_healthcheck_uri`);
  - изменения в конфигурационных файлах.

## 5. Обновления и патчи

### 5.1. Регулярная актуализация

- обновлять:
  - операционную систему (RHEL 8/Alt Linux SP10) до последних версий безопасности;
  - библиотеки и зависимости (например, Ansible 2.9+);
  - IAM Proxy до актуальной версии (через `deploy_platformauth_version`).
- тестировать обновления в тестовых средах перед внедрением в production.

### 5.2. Процедура обновления

- использовать CI/CD-процесс для автоматического применения патчей;
- проводить ручную проверку изменений в `main.yml` и `hosts` после каждого обновления.

## 6. Резервное копирование

### 6.1. Процедура бэкапа

- выполнять ежедневные резервные копии:
  - конфигурационных файлов (`ansible/inventories/StandX/*`);
  - логов аудита (`audit2_options`).
- хранить бэкапы в зашифрованном виде на удаленных хранилищах.

### 6.2. Тестирование восстановления

- проводить тесты восстановления не реже одного раза в квартал;
- проверять работоспособность бэкапов после каждого обновления системы.

## 7. Интеграция с другими системами

### 7.1. Безопасное взаимодействие

- при работе с альтернативными IDP:
  - проверять корректность `oidc_alt_idp_discovery_url`;
  - настроить `oidc_alt_idp_logout_uri` для безопасного выхода (требуется, если провайдер в `oidc_alt_idp_discovery_url`
    не поддерживает отправку url для выхода).

## 8. Чрезвычайные ситуации

### 8.1. План реагирования

- в случае атаки DDoS:
  - блокировать IP-адреса с подозрительной активностью через фаервол;
  - временно переключить трафик на резервный балансировщик (`loadbalancer_for_proxy`).
- в случае утечки данных:
  - отключить соответствующие учетные записи;
  - перегенерировать секреты (пароли, токены).

### 8.2. Резервные каналы связи

- поддерживать резервные каналы для связи с командой DevOps (например, Telegram/Slack);
- готовить чек-лист действий для быстрого восстановления после инцидента.

## 9. Дополнительные рекомендации

- регулярно проводить аудит конфигураций и проверку.
