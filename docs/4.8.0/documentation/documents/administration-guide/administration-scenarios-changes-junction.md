# Изменение параметров ответвления

## Проблематика  
Администраторы сталкиваются с необходимостью обновления параметров существующих ответвлений (junction) IAM Proxy для адаптации к измененным требованиям — например, изменение `junctionPoint`, `serverAddresses`, политики авторизации, настройки mTLS, лимитов или healthcheck. Некорректные или непроверенные изменения могут привести к недоступности сервисов, ошибкам аутентификации, нарушению безопасности или сбоям в балансировке нагрузки.

## Предусловия  
1. **Доступ к среде**:  
   - наличие прав администратора в системе управления конфигурациями или на сервере с IAM Proxy;  
   - доступ к конфигурационным файлам IAM Proxy (локально или через UI);  
   - возможность перезагрузки конфигурации (через API или перезапуск службы).  

2. **Требуемые навыки**:  
   - понимание структуры `proxy_jct_list` и соответствия полей JSON-схеме `rds-api-configuration.schema.json`;  
   - знание влияния ключевых параметров (`transparent`, `https`, `sslCommonName`, `applyJctRequestFilter`) на маршрутизацию и безопасность (подробнее в [Соответствие имен параметров для разных сред и инструментов установки](../installation-guide/params.md));  
   - опыт работы с YAML, JSON и базовыми инструментами редактирования и валидации конфигураций.  

3. **Инструменты**:  
   - текстовый редактор (например, VS Code, Vim);  
   - утилиты проверки YAML/JSON (например, `yamllint`, `jq`, `jsonschema`);  
   - доступ к терминалу или API IAM Proxy для применения и проверки изменений.

## Правила эксплуатации  
1. **Безопасность конфигураций**:  
   - все изменения должны проходить через утвержденный процесс (GitOps, CI/CD или защищенный UI);  
   - запрещено вносить правки напрямую в production без предварительного тестирования в staging-среде.  

2. **Форматы данных**:  
   - конфигурационные файлы должны быть валидными по YAML-схеме и соответствовать ограничениям из `rds-api-configuration.schema.json`;  
   - при ручном редактировании обязательно проверяйте:  
     - корректность отступов и типов значений;  
     - уникальность `junctionPoint` (не должен дублироваться);  
     - соответствие паттернам (например, `junctionPoint` не должен заканчиваться на `/`, `limitRequestsZone` — только буквы, цифры, `_`, `-`).  

3. **Особенности настройки**:  
   - перед изменением проверьте зависимости (например, интеграцию с MONA, SRLS, Vault, RDS API);  
   - при изменении `serverAddresses` убедитесь, что новые адреса доступны, разрешаются через DNS и защищены TLS (если `https: true`);  
   - при использовании `applyJctRequestFilter` убедитесь, что указанные опции существуют в системе (например, `set-header-host-to-backend`, `ssl-sni-on`);  
   - при настройке `healthcheck` проверьте, что endpoint (`path`) возвращает корректные HTTP-статусы.

## Последовательность выполнения

### Способ 1: Через внешнюю систему управления конфигурациями
1. **Открыть UI системы управления конфигурациями**:  
   - выберите артефакт, соответствующий профилю развертывания IAM Proxy для требуемого стенда.

2. **Найти и изменить нужное ответвление**:  
   - в секции `proxy_jct_list` найдите элемент по `junctionName` или `junctionPoint`;  
   - обновите необходимые параметры в соответствии со схемой:

     ```yaml
     proxy_jct_list:
       - junctionName: "Мое приложение"
         junctionPoint: /my-app
         indexUrl: /my-app/home
         https: true
         sslCommonName: ".backend.mycompany.ru"
         transparent: false
         serverAddresses:
           - "new-backend.mycompany.ru:8443"
         applyJctRequestFilter: "set-header-host-to-backend,ssl-sni-on"
         limitRequests: 50
         limitRequestsZone: "shared-api-zone"
         authorizeByRoleTemplate: "App_.*"
         healthcheck:
           enabled: true
           path: /health
           interval: 15000
           timeout: 5000
           fall: 3
           rise: 2
           validStatus: "200,204,302"
     ```

3. **Сохранить и применить изменения**:  
   - подтвердите и сохраните конфигурацию;  
   - дождитесь синхронизации с IAM Proxy или инициируйте перезагрузку конфигурации через API.

### Способ 2: Через YAML-файл (вручную)
1. **Редактировать конфигурационный файл**:  
   - откройте файл `ansible/inventories/<StandX>/group_vars/all/main.yml`;  
   - найдите и обновите нужное ответвление в `proxy_jct_list`, соблюдая синтаксис и ограничения схемы:

     ```yaml
     proxy_jct_list:
       - junctionName: "Обновленное приложение"
         junctionPoint: /updated-app
         serverAddresses:
           - "api-v2.corp.local:443"
         https: true
         transparent: true
         applyJctRequestFilter: "large-upload"
     ```

2. **Применить изменения**:  
   - сохраните файл;  
   - выполните повторное развертывание через Ansible или перезагрузите конфигурацию IAM Proxy:

     ```shell
     curl -X POST https://iam-proxy.mycompany.ru/actuator/reload -k
     ```

## Результат  
Параметры ответвления успешно обновлены. Проверьте:  
- доступность сервиса через IAM Proxy:  
  ```shell
  curl -k https://iam-proxy.mycompany.ru/updated-app/
  ```  
- отсутствие ошибок в логах IAM Proxy (`/opt/iamproxy/logs/access.log`, `error.log`);
- корректность применения изменений (например, через UI IAM Proxy или проверкой заголовков ответа);
- работоспособность healthcheck (если включен) — убедитесь, что бэкенд помечается как `up`.

## Описание механизмов безопасности
1. **Ограничение доступа**:
   - редактирование конфигураций разрешено только пользователям с правами администратора;
   - доступ к UI и API защищен с использованием TLS 1.2+ и аутентификации по OIDC.

2. **Целостность данных**:
   - конфигурационные файлы хранятся в защищенных хранилищах (Vault, SecMan);
   - перед применением изменения проходят валидацию по схеме `rds-api-configuration.schema.json`;
   - пример безопасной конфигурации:

     ```yaml
     proxy_jct_list:
       - junctionName: "Защищенное ответвление"
         junctionPoint: /secure-service
         https: true
         sslCommonName: "*.secure.mycompany.ru"
         serverAddresses:
           - "svc-01.secure.mycompany.ru:8443"
         applyJctRequestFilter: "set-header-host-to-backend,ssl-sni-on"
         authorizeByRoleTemplate: "regex:^SecureAdmin_.*"
     ```

3. **Логирование и аудит**:
   - все изменения конфигурации фиксируются в централизованном журнале (Platform V Audit SE);
   - события изменения `proxy_jct_list` включают: кто изменил, когда, какие поля и с какого IP;
   - рекомендуется настроить оповещения о внесении изменений через MONA.
