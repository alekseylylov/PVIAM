# Часто встречающиеся проблемы и пути их устранения

1. Почему на разных стендах IAM Proxy (ПСИ, ИФТ и прочих) время жизни сессии может отличаться?

   Время жизни сессии может отличаться в зависимости от стендов и окружения, и в основном зависит от заданного на IAM
   Proxy тайм-аута сессии по неактивности и от времени жизни access и refresh токенов. Время истечения access и refresh
   токенов определяется на IDP, используемый для аутентификации.

   На IAM Proxy время жизни сессии по неактивности задается через параметр `proxy_session_idletime` (более подробное
   описание в разделе [Установка](../installation-guide/installation.md)). При отсутствии каких-либо запросов к IAM
   Proxy в течении этого времени приведет к признанию сессии просроченной, и она будет проигнорирована на IAM Proxy (что
   приведет или к переадресации и запуску новой аутентификации или к ошибке аутентификации с HTTP кодом 401).

2. IAM proxy вызывает много раз IDP с попыткой обновить токены, и получает при этом множество ошибок при обновлении
   токенов от IDP.

   Подобное поведение может быть связано с тем, что приложение не обрабатывает должным образом HTTP код 401, получаемый
   в ответах от IAM Proxy при просроченных/невалидных токенах. При этом обычно приложение из браузера создает множество
   запросов завершающихся 401 кодом, на каждом таком запросе IAM Proxy пытается обновить токены и делает запрос в
   сторону IDP. В итоге получаем большое количество запросов от множества браузеров, приводящие к вызову IDP и к
   повышению нагрузки на IDP.

   В Platform V IAM 1.8.0 были реализованы доработки, позволяющие исключить повторные неуспешные запросы к IDP, и сильно
   снизить влияние на IDP при описанных кейсах.

3. Возникла необходимость заниматься маршрутизацией в Istio самостоятельно, в связи с чем необходимо, чтобы порты в IAM
   Proxy передавались без изменений. Как это реализовать?

   При использовании Service Mesh(Istio/SSM) для маршрутизации внутри Namespace создаются уникальные порты для каждого
   сервера из ответвлений. Это необходимо для корректной маршрутизации трафика внутри Namespace, и является необходимым
   при наличии нескольких серверов, и оригинальные порты серверов для внутренней маршрутизации использовать не
   рекомендуется. Поэтому в сервисе IAM Proxy отсутствует опция отключения механизма замены портов.

   > Номера портов можно вычислить, используя такую же логику, как в шаблонах IAM Proxy.
   > Шаблоны IAM Proxy есть в дистрибутиве, в артефакте с конфигурацией `auth-cfg` в папке
   > `package/k8s/base/istio/config/`.

4. Как правильно настроить активный `healthcheck` на сервисе IAM Proxy? Есть ли возможность задать контекст для запроса
   `healthcheck`, отличный от `/healthcheck`?

  - в версии Platform V IAM 2.2.0 реализованы доработки, позволяющие настроить параметры healthcheck для каждого
    ответвления отдельно. Подробнее смотрите описание параметра `proxy_healthcheck_enable` в
    раздела [Установка](installation.md);
  - в версиях до Platform V IAM 2.2.0 определить другой endpoint возможности нет:
    - IAM Proxy отправляет только на контекст `/healthcheck` на используемый бэкенд
      (`https://mycompany.ru/healthcheck`);
    - какие коды HTTP-ответов считаются успешными ответами на запрос healthcheck? Успешными считаются следующие коды
      HTTP ответа: `200`, `204`, `302`;
    - есть ли возможность задать периодичность healthcheck-запросов, отличной от 10 секунд? Нет такой возможности;
    - какое количество неуспешных запросов необходимо для вывода узла из балансировки? Для вывода из пула необходим 1
      неуспешный ответ от бэкенд;
    - какое количество неуспешных запросов необходимо для ввода узла обратно в пул балансировки? Ввод узла в пул
      балансировки происходит после 30 сек и 1 успешного ответа от бэкенд.

5. Как передать несколько вариантов ролей в параметр `authorizeByRoleTemplate`?

  - с версии Platform V IAM 2.1.0 реализованы доработки, позволяющие использовать формат сравнения в виде шаблона
    `regex PCRE`, и можно задать несколько вариантов ролей, например `regex:(role1|role2|admin-.*|admin2)`. Подробнее
    смотрите описание параметра `authorizeByRoleTemplate` в разделе [Установка](installation.md);
  - в версиях до Platform V IAM 2.1.0 значение параметра `authorizeByRoleTemplate` для сравнения используется только
    функция Lua `string.match`, и используется формат регулярного выражения Lua (описание формата приведено
    в [руководстве Lua ](https://www.lua.org/manual/5.4/manual.html#6.4.1)).

6. О чем говорит ошибка "Connection timed out"

    ```log
    1970/01/01 00:00:00 [error] 6035263#6035263: 2438788 upstream timed out (110: Connection timed out) while connecting to upstream
    ```

   Ошибка говорит о том, что, при подключении к бэкенд на уровне tcp/L4, ожидание ответа превысило тайм-аут (на сервисе
   IAM Proxy он по умолчанию равен 3 секундам). Возможно бэкенд недоступен или не справляется с большим количеством
   запросов. Рекомендуется проанализировать наличие сетевого доступа, нагрузку, утилизацию ресурсов.

7. Как можно проверить, что бэкенд действительно не успевает обработать запросы?

   В качестве альтернативного способа диагностики (не используя сервис IAM Proxy) и подтверждения выше описанной теории,
   можно использовать скрипт приведенный ниже. Запустить скрипт необходимо с того сервера или контейнера IAM Proxy, на
   котором возникает проблема с подключением
   (при этом должны быть права на запись в текущую папку, для записи результатов).

   **Параллельная отправка всех запросов**

   ```shell
   #!/bin/bash

   count=250
   url=<backend, стоящий за IAM Proxy, на который IAM Proxy отправляет запросы>

   outputPath=check_async_$count_$(date '+%Y-%m-%d_%H-%M')_$$
   rm -f $outputPath
   mkdir -p -v $outputPath

   for ((i=1; i<=$count; i++ )); do
   curl -v -w @- -o /dev/null -s "$@" $url &>$outputPath/req-$i.log <<EOF &
     ----------\n
     time_namelookup:  %{time_namelookup}\n
     time_connect:  %{time_connect}\n
     time_appconnect:  %{time_appconnect}\n
     time_pretransfer:  %{time_pretransfer}\n
     time_redirect:  %{time_redirect}\n
     time_starttransfer:  %{time_starttransfer}\n
     time_total:  %{time_total}\n
   EOF
   done
   ```

   **Последовательная отправка запросов (по одному, с ожиданием завершением предыдущего)**

   ```shell
   #!/bin/bash

   count=250
   url=<backend, стоящий за IAM Proxy, на который IAM Proxy отправляет запросы>

   outputPath=check_sync_$count_$(date '+%Y-%m-%d_%H-%M')_$$
   rm -f $outputPath
   mkdir -p -v $outputPath

   for ((i=1; i<=$count; i++ )); do
   curl -v -w @- -o /dev/null -s "$@" $url &>$outputPath/req-$i.log <<EOF
   ----------\n
     time_namelookup:  %{time_namelookup}\n
     time_connect:  %{time_connect}\n
     time_appconnect:  %{time_appconnect}\n
     time_pretransfer:  %{time_pretransfer}\n
     time_redirect:  %{time_redirect}\n
     time_starttransfer:  %{time_starttransfer}\n
     time_total:  %{time_total}\n
   EOF
   done
   ```

   После завершения скрипта будет создана папка (формат названия:
   `checkConnection_<количество запросов>_<дата, время>_<случайные 4 символа>`) с результатами выполнения каждого из
   запросов (количество файлов будет равно числу отправленных запросов).

   Можно изменить в скрипте значение параметра `count` на другое (имеет смысл задавать значение не менее 10).
   Рекомендуется провести тест несколько раз, в разное время (в периоды наиболее высокой и наименее высокой нагрузки), к
   примеру, проблема может воспроизводиться только при большом количестве запросов к бэкенд.

   > Внимание!
   >
   > Данный скрипт может одномоментно создать нагрузку до 250 rps. Рекомендуем выполнить его сначала в тестовой среде,
   > а также свериться с проводившимися результатами НТ на предмет превышения допустимых нагрузок.

8. Как увеличить тайм-аут на подключение к бэкенд?

   > По умолчанию тайм-аут на tcp подключения задан в 3 секунды, что в штатной ситуацию вполне достаточно для 
   > подключения на уровне tcp/L4.
   > Если подключение происходит за более длительное время, то скорее всего имеются инфраструктурные проблемы или
   > присутствует деградация бэкенд сервиса (например при исчерпании системных ресурсов), и необходимо выяснить причины
   > такого длительного подключения.

   **Обратите внимание, что данный параметр отвечает за тайм-аут подключения на уровне tcp/L4.**
   Параметр `proxy_connect_timeout` задается только через индивидуальную (custom) конфигурацию. Для этого необходимо в
   папке `/opt/iamproxy/conf/custom.d`, создать файл именем по шаблону
   `<my-name>.location.conf` (например `increase-proxy-connect-timeout.location.conf`), с текстом:

   ```
   proxy_connect_timeout 10s;
   ```

   Вместо 10 задайте необходимое значение в секундах, после чего в параметр ответвления `applyJctRequestFilter` добавить
   значение `custom.d/increase-proxy-connect-timeout.location.conf` (несколько значений указывается через запятую).
   После этого выполните развертывание IAM Proxy, или рестарт контейнера, в зависимости от варианта развертывания.

9. Почему при отправке запроса на IAM Proxy иногда вместо redirect (302) пользователь получает ошибку 401?

   На IAM Proxy есть логика определения API и UI запроса. Один из признаков - значение HTTP заголовка `Accept`. В
   случае, если значением HTTP заголовка `Accept` является значение `text/html` или `application/xhtml`, то запрос
   считается с типом UI и отдается 302 код с перенаправлением (redirect) на IDP (запускает сценарий аутентификации по
   OIDC code-flow).

   В случае отсутствия сессии у пользователя, в зависимости от типа запроса, ответ IAM Proxy будет отличаться:
  - если запрос пришел с UI, то будет возвращен 302 код ответа и пользователь отправиться на IDP;
  - если запрос пришел с API, то будет возвращен 401 код ответа.

   Этим поведением можно управлять используя параметры `proxy_request_to_ui_regex` и `proxy_request_to_api_regex`, про
   их использование смотрите в
   разделе [Соответствие имен параметров для разных сред и инструментов установки](../installation-guide/params.md).

10. О чем может говорить ошибка `unknown "o" variable`?

    ```
       1970/01/01 00:00:00 [emerg] 1#1: unknown "o" variable
       nginx: [emerg] unknown "o" variable
    ```

    Где-то в конфигурации попытались использовать переменную, которая не задана. Использование переменных в nginx
    производится через указание префикса `$`, и там где он не ожидался можно получить подобную ошибку.

    Возможно, в параметр `session_secret` в conf-файле (в профиле установки это параметр `proxy_session_secret` или
    `vault_proxy_session_secret` или в vault-хранилище секретов это `proxy_session_secret`), было передано значение
    содержащее запрещенные символы. Более подробное описание про заполнение `proxy_session_secret` можно прочитать в
    разделе
    [Установка](../installation-guide/installation.md).

    > C версии Platform V IAM 2.1.0 такой проблемы с `proxy_session_secret` не будет, и можно использовать любые
    > печатные символы. Не все параметры расчитаны на наличие символа `$`, и это следует учитывать при заполнении
    > профиля установки, и лучше избегать использования `$` и других спец символов в строковых значениях.

12. Как сервис IAM Proxy понимает, что пользователь аутентифицирован?

    Сессия сохраняется в сессионных cookie `PLATFORM_SESSION*`, дополнительная информация для корректного
    функционирования IAM Proxy:

  - `PLATFORM_SESSION`, `PLATFORM_SESSION_2`, `PLATFORM_SESSION_3` ... - cookie, хранящие в себе информацию о сессии
    пользователя;
  - `PLATFORM_SESSION_ID`. ID сессионной cookie (необязательна, и может отсутствовать для определенных конфигураций IAM
    Proxy).

    > Префикс `PLATFORM_SESSION` можно изменить на другой, и задается параметром `proxy_session_name` профиля установки,
    > подробнее в разделе [Установка](../installation-guide/installation).

13. Почему не работает активный healthcheck в IAM Proxy?

**Проверьте конфигурацию healthcheck**, убедитесь, что на бэкенд-сервиса endpoint /healthcheck существует и доступен.

**Убедитесь в доступности бэкенд-сервисов**, проверьте сетевое соединения между IAM Proxy и бэкенд-сервисами.

**Обновите версию продукта**, возможно проблема связана с устаревшей версией IAM Proxy, обновите IAM Proxy до актуальной
версии.

**Настройка статических IP-адресов в SSM**. Если средствами SSM выдаются IP-адреса при разрешении DNS-имен для
ServiceEntry, задайте в ServiceEntry явно IP-адреса. Пример конфигурации ServiceEntry:
```yaml
 apiVersion: networking.istio.io/v1beta1
 kind: ServiceEntry
 metadata:
   name: ext-mycompany-ru
 spec:
   exportTo:
     - "."
   hosts:
     - "mycompany.ru"
   addresses:
     - 240.x.x.168/32
   ports:
     - number: 8080
       name: http
       protocol: HTTP
   location: MESH_EXTERNAL
   resolution: DNS
```   

14. Как работает IAM Proxy при отключенном активном healthcheck?

По умолчанию активный healthcheck не включен на IAM Proxy, и включать его необходимо только если он действительно нужен.
При этом обеспечение отказоустойчивости будет в основном в соответствии с настройками nginx/syngx заданных по
умолчанию.\
По умолчанию запросы распределяются по серверам циклически (в режиме round-robin). Если при попытке работы с сервером
происходит ошибка, то запрос передается следующему серверу, и так далее до тех пор, пока не будут опробованы все
работающие серверы. Если не удастся получить успешный ответ ни от одного из серверов, то клиенту будет возвращен
результат работы с последним сервером.\
Запрос будет считаться неуспешным и передан следующему серверу:
- если произошла ошибка соединения с сервером;
- если произошел таймаут во время соединения с сервером;
- если произошла ошибка передачи ему запроса или чтения заголовка ответа сервера (в том числе по причине проблем с TLS).
После одной неудачной попытки работы с сервером в течении 10 секунд, сервер будет считаться недоступным на период 10
секунд.

> Если в группе только один сервер, то такой сервер никогда не будет считаться недоступным.

15. Как проверить, соответствует ли IAM Proxy требованиям КБ?

- SSL/TLS Конфигурация:
  - проверка использования только безопасных версий протоколов:
    Убедитесь, что в конфигурации указаны только TLSv1.2 и TLSv1.3.
  ```
   ssl_protocols TLSv1.2 TLSv1.3;
   proxy_ssl_protocols TLSv1.2 TLSv1.3;
  ```   
  - отключение устаревших протоколов: Убедитесь, что TLSv1, TLSv1.1 отсутствуют в конфигурации.
- OCSP Stapling:
  - включите `ssl_stapling on;` в секции server:
  ```
  ssl_stapling on;
  ssl_stapling_verify on;
  ```   
  - убедитесь, что параметр не отключен по умолчанию (`ssl_stapling off;`).
- проверка сертификатов бэкенд-серверов:
  - включите `proxy_ssl_verify on;` в конфигурации прокси:
  ```
  proxy_ssl_verify on;
  ```   
  - убедитесь, что значение `off` не указано в секции `location`;
- запрет использования непривилегированных пользователей:
  - IAM Proxy должен запускаться под пользователем `iamproxy`, а не root или nobody;
  - проверить что пользователь `iamproxy` не суперпользователь, команда `id -u` должна возвращать не `0`;
  - проверьте конфигурацию systemd через `systemctl cat iamproxy`:
  ```
  [Service]
  User=iamproxy
  Group=iamproxy
  ```
- настройка пути к конфигурационным файлам:
  - убедитесь, что пути во всех `include` относительные и указаны корректно:
    - `common/*.conf`
    - `jct/*.conf`
    - `blacklist/*.conf`
    - `mime.types`
- директива `ssl` в listen:
  - убедитесь, что все listen-директивы в секции server содержат флаг ssl:
  ```shell
  listen 443 ssl;
  ```  

16. Почему возникает ошибка SSL при подключении IAM Proxy к бэкенд-сервису?

**Обновление сертификатов**:

- проверьте срок сертификатов IAM Proxy и бэкенд-сервиса (в том числе срок сертификатов ЦС), и если есть просроченные
  то замените сертификаты на актуальные и перезапустите IAM Proxy.
  
**Настройка доверия к ЦС**:

- убедитесь, что на бэкенд-сервисе есть доверие к сертификату IAM Proxy (доверие к ЦС, и возможно какие-то проверки по
  атрибутам сертификата, например по CN);
- убедитесь, что путь к ca-bundle (`trusted_chain.crt.pem`) указан правильно в конфигурации IAM Proxy и файл содержит в
  формате PEM все необходимые сертификаты промежуточных и корневых ЦС используемых в бэкенд-сервисах.
