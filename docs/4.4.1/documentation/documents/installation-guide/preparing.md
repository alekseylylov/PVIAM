# Подготовка окружения

## Подготовка элементов развертывания для VM

**Последовательность действий:**

1. Получить серверы с ОС Linux и с необходимой мощностью (поддерживаемые версии ОС смотрите в технических требованиях),
   и получить доступ до них.
2. Настроить репозитории пакетов (`yum`/`apt`/`pip`), и подключить их на серверах.

**Проверка результата:**

- серверы доступны по SSH;
- успешно выполняется установка пакетов из `yum`/`apt`/`pip`.

## Подготовка элементов развертывания для k8s

**Последовательность действий:**

1. Подготовить кластер k8s (Openshift или DropApp) или получить доступ к существующему.
2. Получить в кластере k8s отдельный namespace для установки компонента IAM Proxy с необходимыми квотами, и получить
   доступ к нему.

**Проверка результата:**

- namespace доступен из консоли администратора k8s;
- успешно выполняется pull необходимых образов.

## Настройка окружения для VM

**Последовательность действий:**

1. Создать репозитории пакетов (`yum`/`apt`/`pip`) или получить доступ к существующим.
2. Обеспечить работу с DevOps-инструментами (Git, Jenkins) и доступ к ним.
3. Обеспечить работу Ansible на сервере установки или Jenkins-агентах (можно использовать tool Jenkins)
4. При необходимости обеспечить наличие сервисов OpenID Connect Provider, HasiCorp Vault, MONA, COTE и обеспечить доступ
   к ним.

**Проверка результата:**

- успешно выполняется установка пакетов из `yum`/`apt`/`pip`;
- команда `ansible-playbook --version` запускается на сервере установки или Jenkins-агентах без ошибок;
- тестовые запросы проходят с серверов до необходимых смежных сервисов.

## Настройка окружения для k8s

**Последовательность действий:**

1. Настроить в namespace получение/pull образов контейнеров, и наличие необходимых образов в registry контейнеров.
2. Обеспечить работу с DevOps-инструментами (Git, Jenkins) и доступ к ним.
3. Обеспечить работу Ansible на Jenkins-агентах (можно использовать tool Jenkins), и работу инструментов установки от
   продукта DOT.
4. При необходимости обеспечить наличие сервисов OpenID Connect Provider, SSM, SRLS, HasiCorp Vault, MONA, COTE и
   обеспечить доступ к ним.

**Проверка результата:**

- успешно выполняется pull необходимых образов;
- jenkins Job установки от продукта DOT успешно запускается в режиме самонастройки;
- тестовые запросы проходят с pod кластера k8s до необходимых смежных сервисов.

## Выпуск и подготовка сертификатов

### При использовании `PKI CA` (HasiCorp Vault)

В случае необходимости использовать `PKI CA` для создания SSL-сертификатов, необходимо получить доступ к PKI, и
использовать реквизиты `PKI CA` при настройке параметров установки продукта (руководствуясь разделами по настройке
параметров).

### При использовании доверенного центра сертификации

Необходимо руководствовать процессом и рекомендациями конкретного центра сертификации, чтобы получить сертификаты. Для
продукта нужны сертификаты в формате PEM, и если предоставляется сертификат в другом формате, необходимо его
конвертировать в PEM (ключ необходим без использования пароля).

Обычно процесс получения сертификата в ЦС выглядит так:

1. Формирование секрета и запроса на сертификат.
2. Отправка запроса в центр сертификации.
3. Получение сертификата от центра сертификации.

### При самостоятельном создании сертификата локально

Для тестовых сред можно создать сертификат локально, используя свой корневой сертификат при выпуске сертификатов
сервисов, или создать самоподписанный сертификат. Часто для этих целей используется утилита `OpenSSL`.

Пример создания самоподписанного сертификата с помощью `openssl` (версии 1.1.1 или выше):

```shell
# fqdn по которому будет обращение к сервису от агента/браузера
PROXY_DNSNAME=test.mycompany.ru
openssl req -x509 -newkey rsa:4096 -sha256 -days 365 -nodes \
 -keyout server.key.pem -out server.crt.pem -subj "/CN=$PROXY_DNSNAME" \
 -addext "subjectAltName=DNS:$PROXY_DNSNAME,DNS:$HOSTNAME"
```

### Дополнительная информация

Для любых способов получения сертификатов необходимо обязательно в subjectAltName указывать fqdn-имена по которым
обращаются к сервису, чтобы можно было обеспечить доверие по fqdn-имени на клиенте.

При использовании SNI при маршрутизации (например на L4 БН в k8s) необходимо чтобы сертификаты на разных сервисах были
различные, и отличались по записи в subjectAltName, по которой проверяется fqdn-имя на клиенте. При использовании одного
сертификата, или сертификатов с wildcard, могут быть проблемы с некорректной маршрутизацией из некоторых
агентов/браузерах (в агентах обычно используются keepalive соединения, привязка которых может производиться по данным из
subjectAltName, и keepalive tcp соединение до одного сервиса может быть ошибочно использовано при подключении к другому,
даже если fqdn-имена у них разные).
